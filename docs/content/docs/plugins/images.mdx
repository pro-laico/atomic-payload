description: "Adds the image and favicon upload collections to your Payload admin: web-ready WebP sizes, a favicon picker field, and an image block for your pages."
---

`@pro-laico/images` gives your Payload admin a place to upload images and favicons. The image collection converts uploads to WebP and generates a standard set of responsive sizes, a separate favicons collection keeps favicons out of your main image library, and the package ships a favicon-picker field and an image block you can use elsewhere. It's a tool the [atomic-payload template](/docs/templates/atomic-payload) and other plugins bring along, but you can add it on its own too.

<Callout type="info">
  `@pro-laico/images` is a Tool package, a building block of the Atomic Payload stack, meant to be used alongside `@pro-laico/core` and the other `@pro-laico/*` packages (most easily via the [atomic-payload template](/docs/templates/atomic-payload)), not on its own.
</Callout>

## Setup

<Tabs items={['Standalone', 'With the atomic-payload template']}>

<Tab value="Standalone">

<Callout type="info">
  `@pro-laico/core` comes bundled with this plugin — it's a dependency, not a separate install. Its cached reads reach Payload through a config you register once in your Next.js instrumentation hook (`registerPayloadConfig`); see [@pro-laico/core → Setup](/docs/plugins/core#setup) to wire it up.
</Callout>

Adding the image collections to your own Payload project.

<Steps>

<Step>
### Add the plugin to your Payload config

```ts
import { buildConfig } from 'payload'
import { imagesPlugin } from '@pro-laico/images'

export default buildConfig({
  plugins: [imagesPlugin({ enabled: true })],
})
```

This registers the **Images** collection (where you upload images, and every upload is converted to WebP and given a set of responsive sizes) and the **Favicons** collection (a separate place for `.ico` favicons). Both appear under an **Assets** group in the admin sidebar. Pass `includeFavicons: false` if you only want the Images collection.
</Step>

<Step>
### Upload your images

Open the **Images** collection in the admin and upload your images. Each one gets an `alt` text field (required, used as the title) and a focal point. Accepted types are PNG, JPEG, WebP, and AVIF.
</Step>

<Step>
### Add blur placeholders (optional)

The plugin does **not** wire up blur placeholders itself, because pnpm doesn't install the optional peer next to this package, so the plugin can't reach it. Install [`@oversightstudio/blur-data-urls`](https://www.npmjs.com/package/@oversightstudio/blur-data-urls) yourself and register it **after** `imagesPlugin`, passing the exported `Images` collection:

```ts
import { buildConfig } from 'payload'
import { imagesPlugin, Images } from '@pro-laico/images'
import { blurDataUrlsPlugin } from '@oversightstudio/blur-data-urls'

export default buildConfig({
  plugins: [
    imagesPlugin({ enabled: true }),
    blurDataUrlsPlugin({
      enabled: true,
      collections: [Images],
      blurOptions: { blur: 18, width: 32, height: 'auto' },
    }),
  ],
})
```

That generates a small blurred placeholder for each upload, which the [image block](#the-image-block) can show while the full image loads.
</Step>

</Steps>

</Tab>

<Tab value="With the atomic-payload template">

The [atomic-payload template](/docs/templates/atomic-payload) already includes `@pro-laico/images`: the **Images** and **Favicons** collections, the blur-placeholder plugin wired against the Images collection, and the image block registered with the other Atomic blocks. You just upload your assets.

<Steps>

<Step>
### Upload images and favicons

Open the **Images** collection and upload your images (each needs `alt` text), and the **Favicons** collection for any `.ico` favicons.
</Step>

<Step>
### Pick a favicon

In the **Site Metadata** global, choose the light and dark favicons for your site. Individual pages can override them in their SEO tab.
</Step>

</Steps>

## Using it in your app

### The favicon field

`FaviconField` is a ready-made upload field that points at the Favicons collection, so you can add a favicon picker to any global or collection. It's what [`@pro-laico/site`](/docs/plugins/site) uses for the site's favicons; add it to your own config the same way:

```ts
import { FaviconField } from '@pro-laico/images'

const SiteMetaData = {
  slug: 'siteMetaData',
  fields: [FaviconField({ name: 'lightFavicon' }), FaviconField({ name: 'darkFavicon' })],
}
```

Pass a `name` (and anything else a Payload upload field accepts) to mount more than one picker. The field always targets the `favicons` collection, so an override can't accidentally re-point it.

### The image block

The package ships an **Image** block (slug `ImageChild`) that lets editors drop an image into [Atomic](/docs/plugins/atomic) content. It isn't registered by `imagesPlugin`. It's one of the default child blocks the Atomic renderer wires up, so in most setups you get it for free. To register it in your own block list, import it from the `@pro-laico/images/blocks/imageChild` subpath:

```ts
import { createImageBlock, Image } from '@pro-laico/images/blocks/imageChild'

// The prebuilt block:
const blocks = [Image]

// …or build your own, prepending/appending fields (e.g. a class-name field):
const customImageBlock = createImageBlock({ prependFields: [], appendFields: [] })
```

The block lets the editor choose the image, a stored size, alt text, and `next/image` options (priority, blur, loading, and more). On the front end it renders through `@pro-laico/atomic/children`, which resolves the chosen image and outputs an optimized `next/image`.

<Callout type="info">
  The Image block is **not** re-exported from the package root. Import `createImageBlock` or `Image` from the `@pro-laico/images/blocks/imageChild` subpath.
</Callout>

## Caching & revalidation

The package ships one cached read, `getCachedImage`, from the `@pro-laico/images/cache` subpath. Give it an image document id (and an optional size name) and it returns that upload's URL, reading Payload once instead of on every render. The [image block](#the-image-block) calls it so the block looks up the chosen image by id, which keeps the rest of the page cached even when the image changes.

The Images collection revalidates the matching tag for you: saving an image refreshes its cached URL, and deleting one clears it, so the next page render serves the new URL. See [Caching & revalidation](/docs/concepts/caching) for how the tags work.

## Options

`imagesPlugin(options?)` accepts:

<TypeTable
  type={{
    enabled: { type: 'boolean', default: 'true', description: 'When false, the plugin does nothing.' },
    includeFavicons: { type: 'boolean', default: 'true', description: 'When true, registers the Favicons collection alongside Images.' },
    imagesOptions: { type: 'Partial<CollectionConfig>', description: "Tweaks merged onto the Images collection, e.g. add fields or change the upload folder. Upload, access, and admin settings are deep-merged and fields/hooks are appended, so a partial override can't silently drop the built-in image sizes, accepted types, alt field, or access rules." },
    faviconsOptions: { type: 'Partial<CollectionConfig>', description: 'Tweaks merged onto the Favicons collection (same merge behavior as imagesOptions).' },
  }}
/>

`imagesOptions` and `faviconsOptions` are standard Payload `Partial<CollectionConfig>` objects, so any collection key you pass is merged onto the built-in collection. All options at their defaults, as a working starting point:

```ts
imagesPlugin({
  enabled: true,
  includeFavicons: true,
  // imagesOptions and faviconsOptions are unset by default (no overrides merged)
})
```

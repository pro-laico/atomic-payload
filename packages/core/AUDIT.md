# @pro-laico/core — Payload Plugin Audit

## Summary
`@pro-laico/core` is structurally solid: `revalidationPlugin` and `jsonSchemaPlugin` are correctly curried and preserve existing config/hooks, `revalidateTag` already uses the non-deprecated `revalidateTag(tag, 'max')` form, and cache getters/loggers are consistently gated with `server-only`. The serious problems are in packaging, not logic: subpath exports (`./cache`, `./admin/*`, `./next/*`, `./logger`) all resolve to a **stale `dist/`** (built 2026-05-16; `src` edited 2026-06-02) whose surface no longer matches `src`, the README documents imports that don't exist, and `core` sits in a **circular workspace dependency** with five sibling packages it claims to break cycles for. Logic-level issues are a `href.ts` null-deref, a likely-invalid `'use server'` on getter modules that export factories, and an `overrideAccess` inconsistency across getters.

### 🟠 High
- [x] **`updateHrefHook` throws when `breadcrumbs` is absent** — _Fixed: guards the array before indexing; returns `previousValue` when breadcrumbs are absent. Dist rebuilt._ — `src/hooks/field/href.ts:4` — `const lastBreadcrumbUrl = data?.breadcrumbs[data?.breadcrumbs?.length - 1]?.url`. The optional chain stops at `data?.breadcrumbs`; the immediate `[...]` index then runs on `undefined` and throws `Cannot read properties of undefined`. On a create where breadcrumbs haven't been populated this hook crashes the request. **Fix:** `data?.breadcrumbs?.[(data?.breadcrumbs?.length ?? 0) - 1]?.url`.
- [x] **Cache getter modules marked `'use server'` while exporting factories / non-async values** — _Fixed: removed `'use server'` from all 16 cache getters (keeping `import 'server-only'`). **Confirmed correct by runtime evidence:** these export sync factories (`createGetCached… = (slug) => async …`), and Next 16/Turbopack rejects `'use server'` modules with non-async exports — "Server Actions must be async functions" (seen via `getAtomicClasses` ← `cache/index` ← `cache/auto` ← `iconSelect.tsx` ← admin importMap). `server-only` is the right guard, and it's satisfied: the cache barrel is imported only by **server** components (`iconSelect` is a `SelectFieldServerComponent`; no `'use client'` file imports it), so it never enters a client graph. The two genuine actions (`revalidateTag.ts`, `triggerVercelDeploy.ts`) keep `'use server'`. NOTE: this only surfaced after rebuilding the stale committed `dist`; always rebuild + reload the admin after touching these._ — `src/utilities/cache/getPage.ts:1`, `getPages.ts:1`, `getForms.ts:1`, `getSitemap.ts:1`, `getAtomicClasses.ts:1` (and others) — A `'use server'` file may only export async functions whose arguments/returns are serializable. These modules export factory functions (`createGetCachedPageByHref`, `createGetCachedPages`, …) that *return functions* and take a `SanitizedConfig` (non-serializable) as the first arg, plus non-async const bindings. Under Next's RSC compiler this is an error/footgun (every export gets wrapped as a server action; passing config across the action boundary fails). These already carry `import 'server-only'`, which is the correct guard. **Fix:** remove the `'use server'` directive from getter modules; keep only `import 'server-only'`. Reserve `'use server'` for true server actions (`utilities/revalidateTag.ts`, `ui/.../triggerVercelDeploy.ts`).
- [x] **README documents a `revalidationLogger` named export that doesn't exist** — _Fixed at the source: `log.ts` now exports `revalidationLogger` both named and default, so `import { revalidationLogger } from '@pro-laico/core/logger'` (README) and the existing default import both work. Dist rebuilt._ — `README.md:47` shows `import { revalidationLogger } from '@pro-laico/core/logger'`, but `src/utilities/log.ts:3` (and `dist/utilities/log.d.ts`) only `export default`. The named import resolves to `undefined`. `src/index.ts:6-11` also refers to "`revalidationLogger` from `./utilities/log`" as if named. **Fix:** either add `export { default as revalidationLogger }` to `log.ts`, or correct the README/comments to `import revalidationLogger from '@pro-laico/core/logger'`.
- [x] **README references `ClassNameField` that is no longer in `src`** — _Fixed: dropped `ClassNameField` from the intro utilities line and the `fields/` table row, noting it lives in `@pro-laico/styles` (the field was intentionally moved there, not deleted)._ — `README.md:16,61` list `ClassNameField` among shipped fields, and `src/index.ts` does not export it (only `slugField`, `StorageTab`, `DevModeField`, `TestPathField`, `UniqueTitleField`). `dist/fields/className.*` only survives because `dist` is stale (see Critical #1). **Fix:** drop `ClassNameField` from the README, or restore the field if its removal was unintentional.

### 🟡 Medium
- [x] **`overrideAccess` is applied inconsistently across getters** — _Resolved by analysis: the difference is INTENTIONAL and load-bearing, NOT standardized. Pages use `overrideAccess: draft` because `read: authenticatedOrPublished` lets published docs through with access enforced; header/footer/iconSet/designSet/shortcutSet are `read: authd` (no published tier), so they must omit it (Local API default `true`) or public SSR would get nothing. Standardizing to `overrideAccess: draft` would break those reads._ — `getPage.ts:30`, `getPages.ts:20`, `getSitemap.ts:22` pass `overrideAccess: draft`, but `getHeader.ts`, `getFooter.ts`, `getIcon.ts`, `getFormSubmissions.ts`, `getDesignSet.ts`, `getShortcutSet.ts` pass nothing. The current behavior is actually *safe* (omitting it means access control runs with no user, so nothing leaks), and `overrideAccess: draft` is intentional so authenticated preview can read drafts — but the inconsistency means draft preview of header/footer/iconSet silently returns nothing if their collection access denies anonymous reads, while pages work. **Fix:** standardize: every draft-aware getter should pass `overrideAccess: draft` (these are server-only, trusted, behind `unstable_cache`), or document explicitly why some collections deliberately do not.
- [x] **`getCachedFormSubmissions` returns submissions with no access override and no auth context** — _Documented: added a JSDoc making explicit that the getter runs with the Local API default and exposes exactly what the host's `form-submissions` `read` access permits — so that collection must restrict reads; this getter adds no gate beyond `server-only`._ — `src/utilities/cache/getFormSubmissions.ts:11-13` — `payload.find({ collection: 'form-submissions', ... })` runs with default access (no user). If the `form-submissions` collection's `read` access permits anonymous reads (common misconfiguration), this cached getter would expose every submission for a form title to anyone who can trigger the cache fill. Not a bug in core per se, but core ships the getter. **Fix:** add an explicit comment that this getter relies entirely on the host collection's `read` access, and consider gating it so it is only callable server-side from trusted paths (it already has `server-only`, which is the relevant guard).
- [x] **`revalidateTag` secondary revalidations are fire-and-forget (not awaited)** — _Fixed: the `pages`→`sitemap` and `designSet`/`shortcutSet`/`atomic-classes`→`site-css` recursions are now awaited._ — `src/utilities/revalidateTag.ts:28-34` — `case 'pages': revalidateTag('sitemap', draft)` and the `designSet`/`shortcutSet`/`atomic-classes` → `revalidateTag('site-css', draft)` recursions are called without `await` and without `return`. Inside a `beforeChange` hook the request may resolve before the secondary `rt()` runs; under Next, `revalidateTag` is synchronous so it usually completes, but relying on that is fragile and the lint-visible "floating promise" is real. **Fix:** `await revalidateTag('sitemap', draft)` (the function is async) and let control fall through, or restructure so the primary tag still revalidates after the secondary.
- [ ] _Deferred — needs careful, tested refactor: extracting a shared `buildTags(tag, tid, draft)` used by both `revalidateTag` and `createGetCached` is the right fix, but it rewrites the tag-string construction on both sides at once — a subtle mismatch would silently break cache invalidation. Worth doing deliberately with verification, not as a drive-by; left for a focused pass. (Current behaviour appears correct; this hardens against future drift.)_ **`revalidateTag` falls through for `'sitemap'`/`'site-css'` secondary tags but those have draft handling that double-suffixes** — `src/utilities/revalidateTag.ts:45-51` — after the `switch`, a `draft` call does `rt(mt([t,'draft']), 'max')`. When the secondary recursion calls `revalidateTag('sitemap', draft=true)`, sitemap is not in `NoDraftTagGroup`, so it also gets a `:draft` suffix. Confirm the getter writes the same key — `getCached` builds `keyParts`/`dependencyTags` independently (`getCached.ts:43-52`); the tag strings must match exactly or revalidation silently misses. There is no shared constant tying revalidate-side and get-side tag construction together. **Fix:** extract a single `buildTags(tag, tid, draft)` helper used by both `revalidateTag` and `createGetCached` to guarantee they agree.
- [x] **`createRevalidateCacheOnDelete` handlers ignore the seed guard** — _Fixed: the delete dispatcher now early-returns on `context.isSeed`, matching the beforeChange one._ — `src/hooks/collection/revalidate.ts:42-48` — the `beforeChange` dispatcher checks `if (context.isSeed) return` (line 30) but the `afterDelete` dispatcher does not, so deleting seeded docs triggers revalidations during seeding. **Fix:** thread `context` into `createRevalidateCacheOnDelete` and short-circuit on `context.isSeed` for symmetry.
- [x] **`pages` delete handler is the only async-unaware handler** — _Fixed: now `async` and awaits both `revalidateTag` calls._ — `src/hooks/collection/revalidate.ts:87-90` — every other handler is `async`/awaits, but `pages: ({ doc }) => { revalidateTag('pages', false); revalidateTag('page', doc?.href, false) }` is sync and fires two un-awaited promises. **Fix:** make it `async ({ doc }) => { await revalidateTag('pages', false); await revalidateTag('page', doc?.href, false) }`.

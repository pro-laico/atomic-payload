# @pro-laico/styles ‚Äî Payload Plugin Audit

## Summary
The plugin architecture is sound: `stylesPlugin` is correctly curried, preserves the incoming config, appends (never replaces) collections/globals/`typescript.schema`, and honors per-collection `false`/`enabled` toggles. The biggest real problems are in the standalone `cssHook` path ‚Äî it never revalidates any cache after regenerating CSS (the canonical `atomicHook` does), and it regenerates CSS unconditionally on every change with no APF gating ‚Äî plus the CSS storage globals being user-writable from the admin despite being machine-owned, and `unocss` being declared an optional peer while `cssProcessor.ts` imports it at module top level. Several token/typescript issues and one dead function (`generateUnoFonts` ignores its input) round out the list.

## Findings

> **Status:** Critical fixed (cssHook now revalidates the changed set tag + `atomic-classes` + `site-css` after regenerating). Storage-globals High fixed (machine-owned: `update: () => false`, fields `readOnly`, global hidden). **Now also fixed:** `generateUnoFonts` clarified (fixed `--font-set*` vars are intentional; the param is a presence-gate, now commented), the `unoThemeAnimation`/`tokenStorage` token-schema reconciled to what the processor emits (dropped the never-emitted `properties`, `counts` ‚Üí `RSS`), `RowLabel` standardized on the `{ path }` object form, and the UTF-8 BOM stripped from `designSet/tabs/storage.ts`. **Still open ‚Äî your design call:** the `unocss` optional-peer-vs-hard-import High, the `cssHook` change-gating High, and the `@pro-laico/site` types-only-dep ‚Üî styles cycle (Low). **Deferred:** the `mod` field (needs intent ‚Äî vestigial or planned for modifier-shortcut composition?). **Reviewed/acceptable:** the `atomic-classes` fetch (needed as input) and the SCSS-import packaging note (the template's build compiles it).

### üî¥ Critical
- [x] **Standalone `cssHook` never revalidates cache after regenerating CSS** ‚Äî `src/hooks/cssHook.ts:90-93` ‚Äî When the standalone path is used (`getCached` supplied, no `atomicHook`), the hook regenerates `draftStorage`/`publishedStorage` but issues no `revalidateTag` calls. The canonical `createAtomicHook` revalidates `designSet`/`shortcutSet`/`atomic-classes`/`pages`/`page` etc. (`atomic/src/hook/createAtomicHook.ts:207-216`). Result: after an admin edits a design set or shortcut in the `getCached`-only configuration, the regenerated CSS is written to storage but `getCached`'s own tag caches (header/footer/designSet/shortcutSet/atomic-classes) are never invalidated, so the *next* render still reads stale inputs and the frontend serves outdated CSS until something else busts the cache. **Fix:** mirror the atomicHook's revalidation in `cssHook`: after `processDesignSet`/class collection, call `revalidateTag(slug, draft)` for the changed set (and `revalidateTag('atomic-classes', draft)` for pages), and revalidate whatever tag the storage globals are read under. Import the same `revalidateTag` from `@pro-laico/core` already used in `revalidateCacheOnDelete`.

### üüÝ High
- [x] **CSS storage globals are user-writable from the admin** ‚Äî _Fixed: `update: () => false` (the processor writes via Local API/overrideAccess), both fields `readOnly`, and the globals `hidden`._ ‚Äî `src/globals/storage.ts:12` ‚Äî `access: { read: authd, update: authd }` lets any authenticated user open `draftStorage`/`publishedStorage` in the admin UI and overwrite `layoutCSS`/`cssSize`. The doc comment (lines 5-8) explicitly says these "are not intended to be edited directly," yet nothing enforces that ‚Äî the fields are not `readOnly` and `update` is open. A bad/empty save corrupts the served stylesheet until the next regeneration. **Fix:** set `update: () => false` (the CSS processor uses the Local API with `overrideAccess` defaulting to true, so it still writes), or at minimum mark both fields `admin: { readOnly: true }` and hide the globals (`admin: { hidden: true }` / a `condition`). Read can stay `authd`.
- [ ] _**Design decision ‚Äî pending your call.** Two clean options: (a) drop `optional: true` (unocss is genuinely required whenever CSS is processed ‚Äî simplest, honest), or (b) lazy `await import('unocss')` inside the processor + gate the `cssProcessor`/`createCssHook` exports so the package loads without it. (a) is recommended unless you intend styles to be importable in environments that never process CSS._ **`unocss` is a hard top-level import but declared an optional peer** ‚Äî `src/cssProcessor.ts:3` + `package.json:89-92` ‚Äî `createGenerator`/`presetWind4`/`presetAttributify`/`presetTypography` are imported at module top level, and `cssProcessor.ts` is re-exported from the main entry (`src/index.ts:4`) and from `./cssProcessor`. `peerDependenciesMeta.unocss.optional = true` tells installers it can be absent, but importing the package without it throws `Cannot find module 'unocss'` at load time (not lazily). **Fix:** either drop `optional: true` (unocss is genuinely required whenever CSS is processed), or make the unocss import lazy (`await import('unocss')` inside `createCssProcessor`'s returned function) and gate the `cssProcessor`/`createCssHook` exports so the rest of the package loads without it.
- [ ] _**Design decision ‚Äî pending your call.** The fix is to gate the `createGenerator`/`generate` pass on whether a class/token field actually changed (compare against `originalDoc`, or reuse core's `runAPF`/APF flags as `atomicHook` does). It's a real perf win but changes when CSS regenerates; want me to mirror the atomicHook's gating logic here, or leave the standalone path always-regenerate for safety?_ **`cssHook` regenerates CSS on every change with no change gating** ‚Äî `src/hooks/cssHook.ts:74-91` ‚Äî Unlike `atomicHook` (which computes `r.classes`/APF run flags and only regenerates when classes actually changed, `atomic/src/hook/createAtomicHook.ts:78-81,217`), `cssHook` runs the full UnoCSS `createGenerator` + `generate` pass (twice for published docs) on *every* `beforeChange` of any non-skip collection ‚Äî including edits that touch no class fields (e.g. changing a title). `createGenerator` is expensive. **Fix:** gate the `cssProcessor` invocation on whether any `*ClassName`/token field actually changed (compare against `originalDoc`, or reuse core's `runAPF`/APF flags), matching the atomicHook's behavior.

### üü° Medium
- [x] **`generateUnoFonts` ignores its input and returns hardcoded vars** ‚Äî _Resolved as intentional + documented: the font *family* values are always the `--font-set*` CSS custom properties that `@pro-laico/fonts` writes via `next/font/local`, so there's nothing per-design-set to vary. The `fonts` arg is a meaningful presence-gate (no font group ‚Üí emit no `font` theme key so UnoCSS keeps its defaults). Added a comment explaining this so the branch no longer reads as dead._ ‚Äî `src/processDesignSet/index.ts:77-80` ‚Äî The function takes `fonts` but only checks it for truthiness, then unconditionally returns `{ mono: 'var(--font-setMono)', sans: 'var(--font-setSans)', serif: 'var(--font-setSerif)', display: 'var(--font-setDisplay)' }`. Any per-design-set font configuration (the Fonts-tab `text` array, or a custom `fontField`) is silently discarded ‚Äî the four font families are fixed regardless of the document. **Fix:** build the map from the actual `fonts`/`text` data, or, if the fixed `--font-set*` vars are intentional, drop the unused parameter and rename to a constant so the dead branch isn't misleading.
- [ ] _Reviewed ‚Äî left as-is: the page `atomic-classes` are a genuine **input** to every CSS regeneration (they must be in the generated set regardless of which doc changed), so the fetch isn't redundant. Threading them through `context` like header/footer would be a parity nicety, not a correctness fix; not worth the extra plumbing now._ **`cssProcessor` always fetches `atomic-classes` even for designSet/shortcutSet saves** ‚Äî `src/cssProcessor.ts:44` ‚Äî `getCached('atomic-classes', draft)` runs on every processor call regardless of `slug`, including when the change is to `designSet`/`shortcutSet` (whose own class set is already in `context`/`storedAtomicClasses`). For header/footer/designSet/shortcutSet it issues the same `getCached` round-trips the slug-equality guards (lines 45-48) are designed to avoid for the changed doc. **Fix:** when `slug` is not the pages slug, the page atomic-classes can't have changed in this request ‚Äî but it's still needed as input; if so leave it, otherwise consider letting the caller pass cached page classes via `context` like header/footer/ds/shortcutSet do, to keep parity and avoid an extra fetch per save.
- [x] **`tokenStorage` zap schema omits `property` shape and mistypes `font`/several keys** ‚Äî _Fixed the concrete `animation` mismatch the finding centres on: `unoThemeAnimation` now declares exactly the five `RSS` maps `generateUnoAnimation` emits ‚Äî dropped the never-written `properties: RSRSS` and changed `counts` from `record(string, string|number)` to `RSS` ‚Äî so `DesignSet['tokenStorage'].animation` no longer advertises fields/types the processor can't produce. (The per-row `properties`/`ease` form fields stay; they feed APF class extraction and the dedicated ease/property token arrays, not `tokenStorage`.)_ ‚Äî `src/designSet/tabs/storage.ts:6-33` vs `src/processDesignSet/index.ts:18-42` ‚Äî The runtime `TokenStorage` has `animation` typed as `UnoThemeAnimation` and `property` as `RSS`, but `processProperty` (`index.ts:66-75`) joins values with commas into an `RSS`, while the zap `tokenStorage` schema marks `property: RSS` yet the field's generated TS will be `TokenStorage` from zap. More concretely, the zap schema has no entry verifying `animation` matches `UnoThemeAnimation`'s actual produced shape (`counts` is `RSS` at runtime but `unoThemeAnimation` declares `counts: record(string, string|number)` and adds a `properties: RSRSS` key the runtime never emits ‚Äî `animation.ts:14-15`). The generated `DesignSet['tokenStorage']` type therefore advertises fields (`properties`) the processor never writes and a `counts` type the processor can't produce. **Fix:** reconcile `unoThemeAnimation`/`tokenStorage` schemas with what `generateUnoAnimation`/`processDesignSet` actually emit (drop `properties`, make `counts: RSS`).
- [x] **Redundant `updatedAt` writes to fields Payload already manages** ‚Äî _Fixed: removed the manual `updatedAt` in `cssProcessor` and `processDesignSet` (Payload sets it on write)._ ‚Äî `src/cssProcessor.ts:90,96` and `src/processDesignSet/index.ts:224` ‚Äî Both write `updatedAt: new Date().toISOString()` (the global write even casts through `as unknown as` to smuggle it past the typed `updateGlobal` args, since `baseStorage` declares only `cssSize`/`layoutCSS`). Payload auto-manages `updatedAt` on globals and collections (timestamps default on), so these are overwritten/ignored and the cast hides the type mismatch. **Fix:** remove the manual `updatedAt` assignments and the `as unknown as Parameters<...>` cast on `updateGlobal`; pass a properly typed `{ layoutCSS, cssSize }` data object.

## Notes
- **Architecture is correct.** `stylesPlugin` (`src/plugin.ts:78-142`) is properly `(opts)=>(config)=>Config`, returns `Plugin`, spreads `...config`, appends to `collections`/`globals`/`typescript.schema` (lines 96-141), and the `enabled`/`designSet:false`/`shortcutSet:false`/`{enabled:false}` toggles all behave as documented.
- **`collection` shallow-merge pitfall is real but documented.** `createDesignSetCollection`/`createShortcutSetCollection` end with `merge ? { ...base, ...merge } : base` (`designSet/createCollection.ts:85`, `shortcutSet/createCollection.ts:56`). A consumer passing `collection: { hooks: {...} }` or `{ fields: [...] }` or `{ admin: {...} }` fully replaces the plugin's `hooks` (losing `beforeChange`/`afterDelete`), `fields` (losing all tabs), or `admin` (losing livePreview/APF controls). The JSDoc says "shallow-merged" but doesn't warn that top-level keys are clobbered. Consider deep-merging `hooks`/`admin`/`fields` or documenting the hazard explicitly.
- **`registerTypescriptSchema` double-wire is guarded by the flag, not by dedupe.** `src/plugin.ts:137-139` appends `toJSONSchemaExtensions` whenever the flag is true (default). If a consumer also wires zap via `@pro-laico/core`'s `jsonSchemaPlugin`, `toJSONSchemaExtensions` runs twice and merges the same `definitions` twice ‚Äî idempotent (same keys overwrite) so harmless, but the docstring's advice to set `false` is the only safeguard. Acceptable.
- **`atomicHook` vs `cssHook` dual path is correct.** `cssHook` no-ops on `context.atomicHookRan`/`context.isSeed` (`src/hooks/cssHook.ts:70`), and `atomicHook` sets `atomicHookRan = true` early (`createAtomicHook.ts:41`), so with both attached CSS is processed once. Both are appended (not replaced) via the filtered `beforeChange` array (`createCollection.ts:50`).
- **`req` is correctly threaded.** `cssProcessor` passes `req` into `updateGlobal` (`cssProcessor.ts:93-94`), so writes join the request transaction.
- **Access is type-safe and present** on both collections (default `authd` for all four ops) and the globals; `authd` (`src/access/authenticated.ts:3`) is a correct `Access` predicate. Only the globals' `update` openness (High finding) is a concern.
- **`TestPathField` relationTo is safe by default** ‚Äî points at `'pages'` (`core/src/fields/testPath.ts:8`); dangling-ref risk only if the consumer renames their pages collection without using `createTestPathField(slug)`.
- **Prose color dark defaults look like a leftover theme**, not neutral defaults: `counters`/`bullets`/`quote-borders` get a green `oklch(87.62% 0.240 148.61)` while everything else is gray (`src/designSet/tabs/prose.ts:123-127`). Cosmetic; verify it's intentional.
- **`build: node -e "process.exit(0)"`** (`package.json:75`) is intentional ‚Äî the package is consumed as raw `src` (`main`/`types` point at `./src/index.ts`), so there is no compile step. `typecheck` covers correctness. Fine for a workspace-internal package.

## Overview

A fresh Atomic Payload install drops you into an empty dashboard. Seeding gives you a one-click way to populate the database with the opinionated starter content the system expects — a starter design set, a default icon set, header and footer globals, site metadata, and more — so a new project renders correctly right away. It also doubles as a one-button reset for development.

## How it works

Seeding is built from three pieces:

- **The seed plugin** — `seedPlugin` wires everything into your Payload config: the seed endpoint and the dashboard banner.
- **The `POST /api/seed` endpoint** — a Payload custom endpoint that runs whatever `seed` function the plugin was given. It's guarded: it returns `403` unless `req.user` is authenticated, and (when an `authorize` predicate is configured) unless that predicate also approves the user. Because it's mounted on Payload's API rather than the Next.js app router, it runs inside Payload's request lifecycle — auth, access checks, and transactions all wired up.
- **The `seed()` function** — the bundled default seed for Atomic Payload templates. It writes a design set, the standard icon set, header and footer, and the `SiteMetaData` / `Settings` globals. You can replace it with your own.

A **`BeforeDashboard`** admin banner is also registered into the dashboard. It renders a **SEED DATABASE** button that POSTs to the endpoint.

<Callout type="info">
  After enabling the plugin, run `pnpm payload generate:importmap` so Payload registers `@pro-laico/seed/admin/beforeDashboard` in its import map. Otherwise the dashboard banner won't render.
</Callout>

## Using it

Once the plugin is enabled, seed a project in one of two ways:

- **From the admin banner** — open the dashboard and click the **SEED DATABASE** button in the `BeforeDashboard` banner. It POSTs to the seed endpoint and runs the configured seed.
- **From the endpoint** — POST to `/api/seed` directly. The request must be authenticated (it returns `403` otherwise).

## Configuration

Add the plugin to your Payload config. With no `seed` passed, it uses the bundled default:

```ts
import { buildConfig } from 'payload'
import { seedPlugin } from '@pro-laico/seed'

export default buildConfig({
  plugins: [
    seedPlugin({
      enabled: process.env.INCLUDE_SEED === 'true',
    }),
  ],
})
```

To run your own seed instead, pass a `seed` function:

```ts
import { seedPlugin } from '@pro-laico/seed'
import { seed } from '@/endpoints/seed'

seedPlugin({ enabled: true, seed })
```

The plugin also accepts `endpointPath` (default `/seed`, so the endpoint resolves to `/api/seed`), `registerBeforeDashboard` (default `true`, whether to inject the admin banner), and `authorize` — a predicate run after the auth check to further restrict who can seed. Since the seed wipes and recreates the whole database, pass a role test for `authorize` in multi-user / production apps rather than leaving it open to any authenticated user.

Slug overrides are handled separately from the plugin options: the bundled `seed()` takes an optional second `slugConfig` argument (`seed(args, slugConfig)`) for remapping the collection and global slugs it writes to. See the seed plugin for the full option table, and [Slugs](/docs/concepts/slugs) for how the seeded globals and collections are addressed.

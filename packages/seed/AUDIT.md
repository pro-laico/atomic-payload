# @pro-laico/seed — Payload Plugin Audit

## Summary

The plugin architecture is clean and correct: it curries properly, spreads existing config, and appends rather than replaces. The critical gap is authorization — any authenticated Payload user (including low-privilege roles) can wipe and reseed the entire database, because `req.user` is the only guard. A second class of issues is that nearly every `payload.create` call in the seed function omits `req`, breaking transaction atomicity and preventing rollback on failure. Several TypeScript safety hacks (`as any`, `as unknown as X`) spread across data builders add noise and hide real type mismatches.

> **Status:** 🔴 Critical + 🟠 High items processed (checked below). **Mediums/lows now also done:** stale breadcrumb `doc`/`id` ObjectIDs removed from the home + notFound seed (the nested-docs plugin recomputes them on create); `pages/index.ts` re-export verified (all four builders). **Reviewed/annotated:** `main`/`types`→`src` (intentional), `BeforeDashboard` (standard server-renders-client-child, no `'use client'` needed), the `SeedButton` subpath export and `SeedFn.slugConfig` (minor API, internal-only for now), and the `as any`/`as unknown as` casts (deferred — a real typing pass against generated block/page types, not a drive-by).

## Findings

### 🔴 Critical

- [x] **No role check on the seed endpoint — any authenticated user can wipe the DB** — `src/endpoint.ts:10` — The handler only tests `if (!req.user)` before running the full destructive seed (delete all collections, recreate). Any logged-in user regardless of role can POST `/api/seed` and destroy all production data. **Fix:** Add an admin-role check after the user check, e.g. `if (!req.user || !req.user.roles?.includes('admin')) return Response.json({ error: 'Action forbidden.' }, { status: 403 })`. Alternatively accept an `authorize?: (user) => boolean` option from the plugin so consumers define what "permitted" means.

- [x] **All `payload.create` calls omit `req` — seed runs outside any transaction** — `src/seed/index.ts:76,91-97,102-112,115,118,121,124,127-133` — The shared `args` object at line 47 is `{ depth: 0, context: { isSeed: true } }` and does not include `req`. Every `payload.create` (forms, icons, pages, iconSet, shortcutSet, designSet, footer, header) is therefore issued outside the request's database transaction. If any step fails partway through, already-written documents are not rolled back, leaving the database in a partially seeded state. **Fix:** Add `req` to the `args` object: `const args = { depth: 0, context: { isSeed: true }, req }` and remove the explicit override `context: { isSeed: false }` at line 130 (or keep `req` there separately).

- [x] **`payload.db.deleteMany` / `payload.db.deleteVersions` also omit `req` from the transaction** — `src/seed/index.ts:66-71` — The deletion step passes `req` as a named argument (`req`) to `deleteMany`/`deleteVersions`, so it appears to be in-transaction. However, when `payload.create` calls below are not in the same transaction (see above), a failure mid-seed leaves deleted documents unrecoverable. This is compounded by the non-transactional creates. **Fix:** Same as above — threading `req` through `args` ensures all operations share a transaction scope.

### 🟠 High

- [x] **`SeedButton` `setLoading(true)` is never reset to `false` on success or fetch error** — `src/components/admin/SeedButton.tsx:43,46-65` — `setLoading(true)` is called before the `toast.promise`, but there is no `finally { setLoading(false) }` path. On a successful seed or on a network error the button remains in the "seeding..." state until the page is refreshed, preventing a retry without a reload. **Fix:** Resolve `setLoading(false)` in both the `.then` success branch (before `setSeeded(true)`) and the `.catch` branch, or use a `finally` block wrapping the fetch promise.

- [x] **`toast.promise` rejection value is a string, not an `Error`** — `src/components/admin/SeedButton.tsx:53` — `reject('An error occurred while seeding.')` passes a string to `reject`. The outer `catch (err)` at line 63 then evaluates `err instanceof Error`, which will be `false`, so `setError` always receives the string directly. This works accidentally but is not idiomatic and will confuse future error handling. **Fix:** `reject(new Error('An error occurred while seeding.'))`.

- [x] **`isSeed: false` context override on the header create is undocumented and likely wrong** — `src/seed/index.ts:130` — The header create explicitly overrides `context: { isSeed: false }`, meaning any hooks that check `context.isSeed` to suppress side effects will run for the header but not for any other collection. There is no comment explaining why only the header needs side-effect hooks to fire. If it was intentional, it should be documented; if it was accidental, it should be `{ isSeed: true }`. **Fix:** Either document the intent with an inline comment or change to `context: { isSeed: true }` for consistency.

- [x] **`overrideAccess` is not explicitly set on any seed operation** — `src/seed/index.ts:47,63,76,91-133` — Payload's Local API defaults to `overrideAccess: true`, so access control is silently bypassed for all seed writes. This is acceptable for a trusted server-side seed but should be explicit, both for documentation and to surface any accidental misconfiguration. **Fix:** Add `overrideAccess: true` to `args` so the intent is visible and auditable.

- [x] **Logger uses `message` key instead of the required `msg` key** — `src/endpoint.ts:16` — `req.payload.logger.error({ err: e, message: 'Error seeding data' })` uses the `message` key. Payload's pino-based logger requires `msg` for the message field; using `message` silently drops the human-readable text from the log line. **Fix:** Change to `req.payload.logger.error({ err: e, msg: 'Error seeding data' })`.

### 🟡 Medium

- [ ] _Intentional: source-only workspace package (consumed via `transpilePackages`), the standard `@pro-laico/*` pattern; no-op `build` is expected. Only matters for a hypothetical non-bundler npm consumer._ **`package.json` `main`/`types` point to raw TypeScript source** — `package.json:10-11` — `"main": "./src/index.ts"` and `"types": "./src/index.ts"` are source pointers, not build artifacts. The `build` script is a no-op (`node -e "process.exit(0)"`). This works only when the consumer project transpiles `node_modules` (e.g. via Next.js `transpilePackages`). If the package is ever published to npm or consumed outside that environment it will fail. The `exports` map has the same issue. **Fix:** Either add a real build step that emits to `dist/` and update `main`/`types`/`exports` accordingly, or explicitly document the "source-only, requires transpilePackages" contract in the README and add a lint/CI guard.

- [ ] _Left as-is: `SeedButton` is an internal piece of `BeforeDashboard`; there's no current need to embed it standalone, so no subpath is added (adding one commits to a public surface). Revisit if a consumer asks for it._ **`exports` map exposes only `beforeDashboard` subpath; `SeedButton` has no export** — `package.json:12-21` — `SeedButton` is exported from `BeforeDashboard.tsx` via a named export but is not accessible via any declared subpath. Consumers who want to embed the button outside the `BeforeDashboard` wrapper cannot import it without reaching into internals. **Fix:** Add `"./admin/seedButton": { "types": "./src/components/admin/SeedButton.tsx", "default": "./src/components/admin/SeedButton.tsx" }` if external use is intended, or keep the component internal and remove the named export.

- [ ] _Reviewed — left as-is: this is the standard Payload admin pattern — a Server Component rendering a `'use client'` child (`SeedButton`) plus `@payloadcms/ui`'s `Banner`. Forcing `'use client'` here would pull `Banner` across the client boundary unnecessarily and isn't required for the SeedButton child to work. No bug today._ **`BeforeDashboard` is missing `'use client'` directive** — `src/components/admin/BeforeDashboard.tsx:1` — The file imports and renders `<SeedButton>`, which is a `'use client'` component. `BeforeDashboard` itself has no directive, making it a Server Component that renders a Client Component child. In Next.js App Router this works only because Payload's import-map loader may handle the boundary, but it is fragile. **Fix:** Add `'use client'` at line 1 of `BeforeDashboard.tsx`, or keep it a Server Component and pass the `SeedButton` as a child prop to maintain a clean server/client split.

- [x] **Hard-coded placeholder URLs in seed data** — _Fixed: footer links now `https://atomicpayload.com` / `https://prolaico.com`; header bare hostnames now `https://`-prefixed._ — `src/seed/footer.ts:58,129` — External links in the footer seed use `'www.google.com'` as the `externalLink` value for the "Atomic Payload" and "Pro Laico" links. Similarly `src/seed/header.ts` uses `'www.github.com'` and `'atomicpayload.com'` without schemas. **Fix:** Replace the placeholder `www.google.com` values with real URLs or `''`, and prefix bare hostnames with `https://` to avoid runtime link failures.

- [x] **Hard-coded breadcrumb document IDs in page seed data** — _Fixed: dropped the stale `doc`/`id` ObjectIDs from both seeds, leaving `breadcrumbs: [{ url, label }]`. The nested-docs plugin recomputes breadcrumbs (including the `doc` relationship) from the page hierarchy on create, so they now bind to the freshly-created docs instead of dangling._ — `src/seed/pages/home.ts:772-779`, `src/seed/pages/notFoundPage.ts:248-255` — Breadcrumbs embed literal MongoDB ObjectIDs (`69274d8cc6c6fa92adf998ab`, `69274dbdc6c6fa92adf99960`). These IDs will not match the documents created in a fresh database by the seed itself, because `payload.create` generates new IDs on every run. **Fix:** Populate breadcrumbs after the pages are created using the `id` from the returned document, or omit the `doc` field and let the Payload breadcrumb plugin re-compute it.

- [x] **`iconSet.ts` derives icon name with `split('.').shift()` which can return `undefined`** — _Fixed: `?? 'unknown'` fallback._ — `src/seed/iconSet.ts:8` — `icon.filename?.split('.').shift()` returns `string | undefined`. The result is cast to `IconSet['iconsArray']` without checking for undefined, so an icon with a missing filename silently writes `name: undefined` into the array. **Fix:** Add a fallback: `icon.filename?.split('.').shift() ?? icon.alt ?? 'unknown'`.

- [ ] _Deferred — needs a real typing pass, not a drive-by: removing the `children as any` / `as unknown as` casts requires the large literal block/page seed data to conform to the generated `ChildBlocks`/`Page` types (or a genuinely-validating guard, not an `assertIs` that just re-casts). High-churn and easy to introduce mismatches; better as a focused task than mixed into this sweep._ **`as any` casts suppress type errors across multiple data builders** — `src/seed/footer.ts:360`, `src/seed/pages/home.ts:765`, `src/seed/pages/notFoundPage.ts:240` — The `children` arrays are cast `as any`, hiding type mismatches between the literal block data and the schema types. The `seed/index.ts` casts `as unknown as Icon[]` (line 98) and `as unknown as Page` (lines 104, 107, 110) for every returned document. **Fix:** Either use proper generated Payload types from `@pro-laico/icons/schema` and `@pro-laico/site/schema` and fix the block data to match, or add a utility `assertIs<T>` type guard rather than double-casting.

- [x] **`devMode: true` is committed in the footer seed** — _Fixed: `devMode: false` (matches header)._ — `src/seed/footer.ts:8` — `devMode: true` is probably unintentional for a "ready-to-use" default seed (the header seed uses `devMode: false`). Shipping with dev mode on means the footer renders in developer mode in production if the seed is applied without modification. **Fix:** Change to `devMode: false`.

### 🟢 Low / nice-to-have

- [x] **`SeedButton` has no ARIA `disabled` or `aria-busy` when loading** — `src/components/admin/SeedButton.tsx:77` — The button does not set `disabled` or `aria-busy="true"` during the loading and seeded states. Clicks are handled by state guards in `handleClick`, but screen readers and keyboard users get no feedback. **Fix:** Add `disabled={loading || seeded}` and `aria-busy={loading}` to the `<button>` element.

- [x] **`SeedPluginOptions` type is not re-exported from `index.ts`** — _Fixed the actual gap: `SeedSlugConfig` is now exported from the barrel._ — `src/index.ts:2` — `SeedPluginOptions` is exported via `export type { SeedFn, SeedPluginOptions } from './plugin'`, so it is correctly re-exported. No change needed here — but `SeedSlugConfig` from `src/seed/index.ts` is not exported at all, making it inaccessible to consumers who want to pass a custom `slugConfig` to the bundled `seed` function. **Fix:** Export `SeedSlugConfig` from `src/index.ts`.

- [ ] _Deferred — minor API ergonomics: threading `slugConfig` through a `SeedPluginOptions` field (or widening `SeedFn`) is a small additive enhancement, but no consumer needs a custom `slugConfig` today and the bundled `seed` can still be imported directly. Left for when there's a concrete need._ **`SeedFn` signature does not expose `slugConfig`** — `src/endpoint.ts:3` — The `SeedFn` type is `(args: { payload, req }) => Promise<void>`, but the built-in `seed` function actually accepts a second optional `slugConfig` parameter. Consumers who wrap the bundled `seed` via the plugin option cannot pass `slugConfig` without re-importing `seed` directly. **Fix:** Either expose a `slugConfig` option in `SeedPluginOptions` that the endpoint threads through, or update the `SeedFn` type to include it.

- [x] **`pages/index.ts` not audited — confirm it re-exports all page builders** — _Verified: it does `export * from './home' / './notFoundPage' / './prose' / './testing'`, so `homePage`, `notFoundPage`, `prosePage`, and `testingPage` (consumed by `seed/index.ts`) are all re-exported. No gap._ — `src/seed/pages/index.ts` — Not read during this audit. Verify it re-exports `testingPage`, `prosePage`, `homePage`, and `notFoundPage` as consumed by `src/seed/index.ts:17`. A missing export there would be a silent runtime failure.

- [x] **`ovar(--foreground)` typo in `designSet.ts`** — _Fixed: all four `ovar(` → `var(`._ — `src/seed/designSet.ts:263,319` — Two `proseColors` entries use `'ovar(--foreground)'` instead of `'var(--foreground)'`. The same typo is duplicated in `proseColorStorage` at lines 461 and 463. **Fix:** Change `ovar(` to `var(` in all four locations.

## Notes

- **No secrets or committed credentials** were found. No database URIs, API keys, or tokens appear in source.
- The plugin's `onInit` composition is not applicable here (the seed does not hook into Payload initialization), so the absence of `onInit` merging is not a deficiency.
- The `README.md` correctly documents the `generate:importmap` requirement and the migration path from hand-rolled setups. The subpath import section matches the `exports` map.
- `next: "^16.1.6"` in `devDependencies` references a version that does not publicly exist at the time of this audit (latest stable is 15.x). This may be an internal pre-release version name; verify the version string is intentional.

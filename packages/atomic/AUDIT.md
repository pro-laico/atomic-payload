# @pro-laico/atomic — Payload Plugin Audit

## Summary
`@pro-laico/atomic` is the all-in-one runtime composed of four well-behaved curried sub-plugins (actions, children, forms, hook) that correctly preserve incoming config and append (never replace). The `atomicHook` `beforeChange` orchestrator threads `req` through nested ops and guards loops via `context`, but has correctness gaps (silent error swallowing in `unsetActive`, draft re-render running CSS twice). The most serious problems are in forms security (the public `/api/form-submissions` REST endpoint bypasses the entire server-action SVR pipeline; in-memory rate-limiting is ineffective on serverless; no honeypot/CAPTCHA; client-supplied consent governs PII collection) and packaging (a no-op `build` plus a months-stale `dist/` that two published subpath exports point at, referencing deleted packages).

> **Status:** Forms security pass done — fixed: the public-endpoint bypass (now Local API write + `create: () => false` on submissions, set in the template's `formSubmissionOverrides`), the shared-singleton processor (new per call), client-trusted consent (now read from a server-side `cookiePreferences` cookie), and `unsetActive` error-swallowing (rethrows + logs). The in-memory rate-limit limitation is now documented in code; a shared-store rewrite, honeypot/CAPTCHA, and the no-op-build/stale-`dist` (Critical #2) are deferred — `dist` is tracked under the "Rebuild stale dist" task. **Now also fixed:** `getServerSideURL` warns in production when no server URL is configured (instead of silently POSTing to localhost), and `engines.node` bumped to `>=20` to match the real `next@16`/React-19 floor. **Reviewed but deliberately not changed:** the CSS double-pass can't simply be `Promise.all`'d (both passes write through the same `req` transaction, so concurrent writes aren't safe — the real win is change-detection to skip the draft pass), and the `formProcessor` `as any` is the classic correlated-union limitation in a security-sensitive validated path (proper fix is a discriminated switch, not a drive-by cast swap).

## Findings

### 🔴 Critical
- [x] **Public form-submissions endpoint bypasses the entire SVR pipeline** — `src/forms/submitForm/serverFunction.ts:30-35` — All rate-limiting, sanitation, and validation run inside the `submitForm` server action; the action then POSTs the result to `POST {serverURL}/api/form-submissions`. That REST endpoint is created by `@payloadcms/plugin-form-builder` with public `create` access (`formBuilder.ts` does not override submission `access`). An attacker can POST directly to `/api/form-submissions` with `{ form, submissionData }` and skip rate-limiting, validation, uniqueness checks, and sanitation entirely — flooding submissions and storing arbitrary data. **Fix:** Do not round-trip through the public REST API. Write the submission with the Local API inside the server action (`req.payload.create({ collection: 'form-submissions', data, overrideAccess: true })`), and lock the submission collection's `create` access to `() => false` (template `formSubmissionOverrides.access`) so the only write path is the validated server action. If REST must stay open, move rate-limit/honeypot/validation into a `beforeChange`/`beforeValidate` hook on the submissions collection so they run regardless of entry point.
- [x] **`build` script is a no-op but `package.json` ships `dist` and points exports at it** — _Fixed via option (a): `./hook/client` now resolves to `src` like the rest of the package, the stale `dist/` was `git rm`'d, `dist` removed from `files`, and a package `.gitignore` prevents re-committing build output. (`./hook/light` was later removed entirely — its only purpose was re-exporting `manualLogger`/`sanitizeData`, which now live in `@pro-laico/core`.)_ — `package.json:196` (`"build": "node -e \"process.exit(0)\""`), `package.json:69-76`, `files:190-194` — `./hook/light` → `./dist/hook/light.js` and `./hook/client` → `./dist/hook/frontEnd/atomicStore/index.js` are the only two exports resolving to `dist`; everything else resolves to raw `./src`. Because `build` does nothing, `dist` is never regenerated. The committed `dist/` is from 2026-05-15 and imports packages that no longer exist (`dist/hook/cssProcessor.js:1` `import { deepMerge } from '@pro-laico/ap-apf'`, `:3` `@pro-laico/design-sets/...`). Consumers of `@pro-laico/atomic/hook/light` or `/hook/client` get stale code that fails to resolve. **Fix:** Either (a) point `./hook/light` at `./src/hook/light.ts` and `./hook/client` at `./src/hook/frontEnd/atomicStore/index.tsx` to match the rest of the package and delete the stale `dist`, or (b) implement a real build (swc + tsc as in the plugin template) and make every export resolve to `dist`. Do not mix. Remove `dist` from `files` until a real build exists.

### 🟠 High
- [ ] _**Design decision — pending your call (the atomic shared-store rate-limiting item).** Code limitation is documented; the real fix is a shared store keyed by `backendFormID`+`ip` — Redis/Upstash, or a Payload collection. Which backing store do you want (or is the documented single-instance limitation acceptable for now)? Partially addressed: the per-instance limitation is now documented in code._ — `src/forms/submitForm/form/rateLimiting/simpleSlidingWindow/serverFunction.ts:5-6` — `lastCleanup`/`memoryStore` are module-level objects. On Vercel/serverless or any multi-instance deployment each Lambda has its own store, so an attacker hitting different instances is never limited; instances also reset on cold start. Combined with the public REST bypass above, rate limiting is effectively decorative. **Fix:** Back the sliding window with a shared store (Redis/Upstash, or a Payload collection keyed by `backendFormID`+`ip`). At minimum document the single-instance limitation in the block description.
- [x] **`SubmitFormProcessor` is a mutable module-level singleton shared across concurrent requests** — `src/forms/submitForm/formProcessor.ts:219-223` (`const submitFormProcessor = new SubmitFormProcessor()`; `getSubmitFormProcessor` always returns it) — Instance fields (`this.args`, `this.success`, `this.CCPreferences`, `this.formInputBlocks`, …) hold per-submission state. `process()` is `async` and `await`s between mutations, so two overlapping submissions interleave: request B's `preProcess`/`reset` clobbers request A's `this.args`/`this.success` mid-flight, producing cross-request data corruption or wrong success/failure decisions (a security-relevant failure for validation). **Fix:** Instantiate `new SubmitFormProcessor()` per call, or refactor to pass all state through arguments/locals (no instance fields). The same applies to `ActionBlockStorageProcessor` instances — but those are correctly `new`-ed per hook invocation (`createAtomicHook.ts:95`), so only the form processor needs the fix.
- [x] **PII collection is governed by client-supplied consent, which is trivially spoofed** — `src/forms/submitForm/formProcessor.ts:177-186` — Whether IP/userAgent/referer/timezone/screen size are stored is decided by `this.CCPreferences` (`= clientData.preferences`, `formProcessor.ts:84`), which comes straight from the browser (`useForm.ts:90-95` builds `clientData` from `context.atomicStore.preferences`). A client that always sends `preferences.userData = true` causes IP/UA storage regardless of the user's real consent; conversely a malicious client can suppress it. Consent must be enforced server-side. **Fix:** Read consent from the server-side consent cookie via `headers`/`cookies()` rather than trusting the posted `clientData.preferences`; treat `clientData` as untrusted UI hints only.
- [ ] _**Design decision — pending your call (the atomic honeypot/CAPTCHA item).** Options: a honeypot field check (cheap, no deps), an `Origin`/`Referer` check against `serverURL`, and/or a CAPTCHA/Turnstile SVR block (needs a provider choice + keys). Which do you want, and should they be SVR blocks editors can toggle per form? Now that the public REST bypass is closed (Critical #1), the server action is the only write path, so these would live in the SVR pipeline._ **No spam protection: no honeypot, CAPTCHA, or origin/CSRF check on the submission action** — `src/forms/submitForm/serverFunction.ts` (whole file) — The audit rubric calls for honeypot/spam handling; there is none in the package (`grep honeypot|captcha|csrf` finds nothing under `src/`). The server action accepts any `FormData`. Because it's a server action it has Next's action-ID protection, but the underlying `/api/form-submissions` endpoint has no such protection. **Fix:** Add a honeypot field check (reject if a hidden field is filled), optionally a CAPTCHA/Turnstile validation block in the SVR pipeline, and verify `Origin`/`Referer` matches `serverURL` before storing.
- [x] **`unsetActive` swallows errors and silently returns `undefined` on failure** — `src/hook/unsetActive.ts:22-24` — The `catch` logs via `manualLogger` (only when `process.env.LOGS === 'true'`) and returns nothing. Inside a `beforeChange` running in a transaction, a failed `update` here means the previously-active document is NOT unset, but the current save proceeds and is marked active — yielding two active docs (header/footer/designSet are supposed to be singletons). The caller (`createAtomicHook.ts:84-92`) only uses the return value to trigger cleanup flags and treats `undefined` as "nothing to clean". **Fix:** Rethrow (or return an error sentinel the hook checks) so the failure aborts the transaction, and use `payload.logger.error({ msg, err })` instead of the env-gated `manualLogger`.

### 🟡 Medium
- [ ] _Deferred — `Promise.all` is NOT a safe fix here: both passes write through `req.payload.updateGlobal({ req })`, i.e. the same request DB transaction, and most adapters can't run concurrent queries on one transaction (Postgres "command already in progress", Mongo single-session). The sound optimization is skipping the draft pass when the draft CSS is unchanged (change-detection), which is a larger change — left for a focused perf pass._ **CSS processor runs twice on every non-draft save** — `src/hook/createAtomicHook.ts:217-221` — When `r.classes` and not skipped: `await cssProcessor({ draft: true })` then, if `!draft`, `await cssProcessor({ draft: false })`. Each `cssProcessor` call runs a full UnoCSS generation (`createGenerator`, multiple `getCached` reads). Publishing a page therefore pays two full UnoCSS passes serially on the critical save path. **Fix:** Run the two passes in parallel (`await Promise.all([...])`) where the draft/published outputs are independent, or skip the draft pass on publish if the draft CSS is unchanged.
- [ ] _Deferred — low value, touches the init path: the WeakMap is functionally harmless (the per-hook `inner` memo already prevents repeat imports), so removing it is pure cleanup with no behavioural win and a small risk on a load-bearing lazy-init path. Left until there's a reason to touch that file._ **`atomicHookFactory` WeakMap cache keyed by config object identity rarely hits** — `src/hook/atomicHookFactory.ts:15,31-38` — `cache = new WeakMap<AtomicHookSlugConfig, ...>` keys on the `slugConfig` object reference. Callers typically build a fresh object literal per `atomicHookWith({...})` call, so the cache almost never hits and each call creates a new lazy hook (and a new `buildHook` import on first use). Harmless functionally but the cache provides ~no benefit. **Fix:** Either drop the WeakMap (the per-hook `inner` memoization already prevents repeat imports per hook instance) or key on a stable serialization of the slug config.
- [ ] _Deferred — needs a failure-semantics decision: wrapping the CSS/traversal sections in try/catch is easy, but the real question is whether a CSS-generation failure should **abort** the save or let it **proceed with stale stored CSS** (the audit notes the latter is usually safer for editors). That's a product call with correctness implications, so not a silent drive-by; pairs with the logging cleanup below._ **Error from any `beforeChange` pass aborts the save with an opaque message** — `src/hook/createAtomicHook.ts:169` — `throw new Error('Processed Node Was Set To Undefined Or Null at path: …')` inside the `traverse(...).after` callback. A single malformed node blocks the entire document save with no recovery and no `payload.logger` entry. CSS/UnoCSS generation errors in `cssProcessor` similarly propagate and block saves. **Fix:** Wrap the CSS-processing and traversal sections in try/catch, log via `payload.logger.error`, and decide whether to fail the save or proceed with stale stored CSS (the latter is usually safer for content editors).
- [ ] _Partially addressed / deferred: the load-bearing case (`unsetActive`) already moved to `payload.logger.error`. The remaining `console.*` sites are pure server actions / render helpers with no `req.payload` in scope (`serverFunction.ts`, `SSRProps.ts`), so a clean fix means introducing a shared logger or threading payload — a broader change worth doing alongside the error-handling decision above, not piecemeal._ **`atomicHook` uses `console.error` and form processor uses `console.error`, not `payload.logger`** — `src/forms/submitForm/serverFunction.ts:40` (`console.error(err)`), `src/children/render/renderChildren.tsx` & `SSRProps.ts:351-352` (`console.warn`/`console.log`) — Rubric item 9: server-side logging should use `payload.logger` for structured, level-aware output. Server-action code has no `payload` instance handy, but the hook does. **Fix:** In hook/server contexts where `req.payload` is available, use `req.payload.logger.error({ msg, err })`. For pure server actions, at least standardize on a single logger and avoid `console.log` for errors.
- [x] **`manualLogger` level matching is case-inconsistent and env-gated** — _Casing fixed: tags now matched case-insensitively (upper-cased before compare), so `[ERROR]`/`[STORE]` colorize correctly. The env-gating + routing real errors through `payload.logger` is a separate broader change (the load-bearing one — `unsetActive` — already moved to `payload.logger.error`)._ — `src/hook/utilities/manualLogger.ts:3-10` — Colors check `[Warning]`/`[Error]`/`[Preview]` (title-case) but callers emit `[ERROR]`/`[INFO]`/`[UPDATE]`/`[STORE]` (upper-case) — e.g. `unsetActive.ts:23` emits `[ERROR]` which never matches `[Error]`, so errors print uncolored. All output is gated behind `process.env.LOGS === 'true'`, so production has no store/error visibility at all. **Fix:** Normalize tag casing (match what callers emit) and route real errors through `payload.logger` so they aren't silently dropped when `LOGS` is unset.
- [x] **`runAPF` `active` branch is dead code** — _Fixed (in core): collapsed to the single `Boolean(context[...])` return with an honest comment. Kept current behavior rather than implementing the "active-only" semantics, which would risk the atomicHook flow._ — `core/src/apf/utilities/runAPF.ts:7-8` — Both the `if (… data.active)` and `else` branches `return Boolean(context[\`${id}-${apf}\`])` — identical. The doc comment claims active docs are handled specially; they aren't. This drives every `runAPF` decision in `createAtomicHook.ts:57-65`, so the intended "only run for active docs" semantics silently don't exist. (Lives in `core` but is load-bearing for the atomicHook.) **Fix:** Implement the intended branch or delete the conditional and the misleading comment.
- [ ] _Reviewed — benign, left as-is: the audit itself concludes the `async` arrow const satisfies the directive and the module-scoped `defaultInner` only memoizes a **stateless** hook (no per-request state, unlike the form-processor singleton that was fixed). No correctness issue today; a relocation into `atomicHookFactory.ts` is optional polish if a future Next version complains._ **`'use server'` on `lazyAtomicHook.ts` constrains the module but the exported value is a const, not a function declaration** — `src/hook/lazyAtomicHook.ts:1,25` — The file is marked `'use server'`; the only export `atomicHook` is an `async` arrow assigned to a `const`. Next's Server Actions bundler requires every export to be an async function and the directive's own comment (`:18-21`) acknowledges this. An `async` arrow const generally satisfies it, but `let defaultInner` (`:23`) is module-scoped mutable state in a `'use server'` module — flagged by some Next versions and a shared-state hazard across requests (same pattern as the form-processor singleton, though here it only memoizes a stateless hook so it's benign). **Fix:** Keep the memo but document the intent; if Next flags it, move the memoization into `atomicHookFactory.ts` (no `'use server'`) and re-export.

### 🟢 Low / nice-to-have

## Notes
- **Plugin architecture (rubric 1):** All four sub-plugins follow the curried `(opts) => (config) => Config` shape, return `Plugin`, spread `...config`, and *prepend/append* to `config.blocks` / `config.collections` without replacing — correct. `enabled === false` returns the untouched config (no schema divergence concern here since these add `blocks`, not collections). The documented composition order (forms → actions → children so children stay first) is consistent across the three block-prepending plugins. `atomicHookPlugin` (`hook/plugin.ts:18-25`) correctly clones each collection, clones `hooks`, and **appends** `atomicHook` to `beforeChange` preserving existing hooks. None of the four compose `onInit` — none need to. Good adherence overall; the lazy-import pattern in `lazyAtomicHook`/`atomicHookFactory` correctly keeps UnoCSS out of config-load (dynamic `import('@pro-laico/core/cache/auto')` + `import('@pro-laico/atomic/actions/processor')` on first invocation).
- **atomicHook correctness (rubric 2):** Loop guards are sound — `context.isSeed` early-return (`createAtomicHook.ts:37`), `context.atomicHookRan` set so a standalone `cssHook` stays inert (`:39-41`), and the `runAPF` `context[\`${id}-${apf}\`]` flags set by core's field hooks (`core/.../apf.ts`) before the collection hook runs. `req` is threaded into `unsetActive` (`:84`) and `cssProcessor` (`:219-220`) and into `req.payload.update` (`unsetActive.ts:9-16`) — transaction-safe. On **create** `originalDoc.id` is `undefined`, so the APF field hooks don't set flags and most passes short-circuit; the create path relies on `isActive`/`runSlug` instead — works but is subtle and undocumented. No `new Date()` non-determinism. The double CSS pass (Medium) and error-throw-blocks-save (Medium) are the real concerns.
- **Forms (rubric 3):** Sanitation/validation/rate-limiting are implemented as composable SVR blocks and run in a fixed order (`formProcessor.ts:105-111`). Input sanitation uses `escapeRegex` correctly (`trimText/serverFunction.ts:60-62`) — no ReDoS/injection there. The pipeline itself is reasonable; the problems are that it can be bypassed (Critical #1), the rate-limit store is per-instance (High), consent is client-trusted (High), there's no honeypot (High), and the processor is a shared singleton (High).
- **Actions/children (rubric 4):** Block fields use proper Payload field types with `required`/`condition`; `backendForm`/`formName` are `text` (not relationships), so no dangling `relationTo` refs. The one `dangerouslySetInnerHTML` in this package (`children/render/SSRProps.ts:377`, SVGChild `contents`) renders admin-authored SVG markup raw — an XSS vector if non-trusted users can edit SVG content, but the source/ownership is `@pro-laico/icons` (covered in that package's audit). Client/server boundaries are clean: `.client.tsx` components carry `'use client'`, server renderers carry `'use server'` + `import 'server-only'` (`renderChildren.tsx:1-2`).
- **Access control (rubric 5):** This package performs no Local API reads/writes with user context except `unsetActive` (system op, correctly `overrideAccess` default / threaded `req`). The data-leak risk is entirely the public submission endpoint (Critical #1). `forms` (backend forms) collection is correctly `read/update: authd` in the template; the **submissions** collection is the gap.
- **Packaging (rubric 6):** The exports map is large but mostly consistent (raw `.ts`/`.tsx` via `default`/`types`, source-only). Two exceptions point at stale `dist` (Critical #2). No circular-dep issues observed in the source graph; the README documents the deliberate self-import via subpaths (`@pro-laico/atomic/...`) to break init cycles, and `forms/index.ts:4` documents why `submitForm` isn't barrel-exported. `type: module` set; peer vs dep split is reasonable (`payload`, `next`, `react`, `unocss`, `zustand` as peers). `traverse` correctly a dep.
- **Performance (rubric 8):** Main cost is the per-save UnoCSS generation (Medium) and full-document `traverse` walks (two passes when actions+classes both run — `createAtomicHook.ts:100-104` then `:106-172`). `getCached` is used throughout the read paths, mitigating N+1.

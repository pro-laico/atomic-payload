`@pro-laico/zap` is the shared schema layer that keeps Atomic Payload type-safe. Built on [Zod](https://zod.dev/), it lets each plugin describe its data once and have those shapes flow through to your generated Payload types. Like `core`, it's mostly behind-the-scenes plumbing — you'll rarely reach for it directly.

## Usage

`zap` re-exports Zod with a `z.ap` registry attached. Register a schema under a known ID, then retrieve it — or its type — elsewhere:

```ts
import { z } from '@pro-laico/zap'

// Register a schema
export const MySlug = z.ap.add(z.enum(['a', 'b']), { id: 'MySlug' })

// Retrieve it elsewhere
const slug = z.ap.get('MySlug')

// Use the type alias
const value: z.ap.Type<'MySlug'> = 'a'
```

## The `AtomicRegistry` type

The TypeScript side of the registry lives in `@pro-laico/zap/schema` as a **type alias** — not an interface. It routes through the `@pro-laico/core` kernel:

```ts
// @pro-laico/zap/schema
import type { Get } from '@pro-laico/core'

export type AtomicRegistry = Get<'AtomicRegistry', Record<string, any>>
```

<Callout type="warn">
  Because `AtomicRegistry` is a `type` alias, you **cannot** declaration-merge it — interface merging does not apply to type aliases. There is no `declare module '@pro-laico/zap/schema'` augmentation. `Get<'AtomicRegistry', …>` looks up the `AtomicRegistry` key on the single `PayloadAugment` interface in `@pro-laico/core`, falling back to `Record<string, any>` when unaugmented.
</Callout>

### How registration actually works

Concrete types flow in through Payload's type generation, not a hand-written interface:

1. **Register at runtime** — each package calls `z.ap.add(schema, { id })` to publish a Zod schema under a string ID.
2. **Emit JSON Schema** — `toJSONSchemaExtensions` (via `ap.toJSONSchema()`) aggregates every registered schema into one `AtomicRegistry` definition and feeds it to Payload's `typescript.schema`.
3. **Payload generates types** — it writes `export interface AtomicRegistry { … }` into the project's `payload-types.ts`, one member per registered ID.
4. **The kernel is augmented once** — the generated `payload-types.augment.d.ts` augments `@pro-laico/core` (not `zap`):

   ```ts
   // src/payload-types.augment.d.ts (auto-generated — do not edit by hand)
   import type * as G from './payload-types'

   declare module '@pro-laico/core' {
     interface PayloadAugment {
       AtomicRegistry: G.AtomicRegistry
     }
   }
   ```

5. **Stubs resolve** — `zap`'s `Get<'AtomicRegistry', …>` now resolves to the project's generated shape, so `z.ap.get(id)` and `z.ap.Type<id>` are concretely typed.

To register your own schema in an app, add it at runtime and re-run type generation — you do not (and cannot) declare-merge `AtomicRegistry`:

```ts
import { z } from '@pro-laico/zap'

// The ID becomes a member of AtomicRegistry after `pnpm generate:types`.
export const MySlug = z.ap.add(z.enum(['a', 'b']), { id: 'MySlug' })
```

## Exports

**Subpath imports**

| Subpath | What's there |
| --- | --- |
| `@pro-laico/zap` | Zod re-exported with the `z.ap` registry helper attached |
| `@pro-laico/zap/schema` | The `AtomicRegistry` type alias (resolves through the `@pro-laico/core` kernel) |
| `@pro-laico/zap/src` | The package source entry |

**Key named exports**

- **`z`** (default + named) — Zod, re-exported, with the `z.ap` (`AtomicPayloadZodClass`) registry attached: `add`, `get`, `type`, and `toJSONSchema`
- **`AtomicPayloadZodClass`** — the class behind `z.ap`, exported for typing/extension. The shared singleton is `z.ap`; you normally don't instantiate it yourself.
- **`toJSONSchemaExtensions`** — the Payload-integration helper. Spread it into your `typescript.schema` extension to add every registered schema (via `ap.toJSONSchema()`) to the generated JSON schema's `definitions`:

  ```ts
  // payload.config.ts
  import { toJSONSchemaExtensions } from '@pro-laico/zap'

  typescript: {
    schema: [({ jsonSchema }) => toJSONSchemaExtensions({ jsonSchema })],
  }
  ```

- **`generateBlocksType`** — builds a JSON-schema entry describing a `oneOf` union of block `$ref`s, used alongside `toJSONSchemaExtensions` in a `typescript.schema` function:

  ```ts
  import { generateBlocksType } from '@pro-laico/zap'

  generateBlocksType({ name: 'MyBlocks', refs: ['HeroBlock', 'CtaBlock'] })
  ```

- **`GenerateBlocksTypeProps`** (type) — the argument shape for `generateBlocksType`: `{ name: string; refs: (string | undefined)[] }`
- **`AtomicRegistry`** (type, from `/schema`) — the `Get<'AtomicRegistry', …>` alias that resolves to a project's generated registry shape

## Gotchas

<Callout type="warn">
  **Pinned to `zod@4.1.11`.** There's a known regression on newer versions that breaks handling of registered schemas. Do not upgrade Zod here without verifying.
</Callout>

<Callout type="warn">
  **`server-only`.** The registry imports `server-only` and is populated at config-resolution time on the server — do not import `z.ap` from client components.
</Callout>

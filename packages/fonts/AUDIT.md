# @pro-laico/fonts ‚Äî Payload Plugin Audit

## Summary

The package provides a `Font` upload collection, an optional `fontSet` global, shared `fontUploadField`/`fontUploadFields` helpers, and a `runDownloadFonts` CLI that authenticates against a live Payload instance to pull font files to disk for `next/font/local`. The plugin architecture is largely correct (curried, non-destructive spread, `enabled` guard), but has a documented shallow-merge hazard for `fontOverride` that it warns about without fixing. The download script has one material correctness bug (fonts directory wiped before env validation completes) and several packaging issues that would break consumers using the published dist.

---

## Findings

> **Status:** The wipe-before-guard Critical + the deep-merge / subdirectory-unlink Highs are fixed (dist rebuilt). The exports/`main`/`types`‚Üí`src` items (Critical #2 + High) are the intentional workspace-only pattern, not changed (the redundant `./src` alias was removed). Shared `authd` (now `src/access/authd.ts`, with a JSDoc on overriding access for the role-check item), `timestamps: true`, slot `filterOptions`, configurable `cssVariablePrefix`, and the README tsx/dev-callout are all fixed (dist rebuilt). Deferred as publish-only or consumer-dependent: `PayloadSDK<any>`/token-cast (generic CLI ‚Äî needs the consumer's generated types) and the `moduleResolution` build setting. The eslint-disable item is a false positive (the template already places the comment directly above the import).

### üî¥ Critical

- [x] **Fonts directory wiped before env-var guard fires** ‚Äî _Fixed: the `hasWarnings` guard now runs before `ensureFontOutputs()`, so a misconfig leaves existing fonts untouched._ ‚Äî `src/scripts/downloadFonts.ts:161-166` ‚Äî `ensureFontOutputs()` is called at line 161, which deletes every existing file in `FONT_FILES_DIR` (line 117) and writes an empty definition. Only then does the function check `hasWarnings` and return early (line 163). If any required env var is missing, the old fonts are already gone and the build continues font-less. **Fix:** Move the `if (hasWarnings) return` block to immediately after the `requiredEnvVars` loop (before `ensureFontOutputs()`), so nothing is written to disk until the configuration is known to be valid.

- [ ] _Intentional: source-only workspace consumption (`.`/`./schema` resolve to `src`), the standard `@pro-laico/*` pattern. Only the genuinely-compiled surfaces (the `bin` and `./scripts/downloadFonts`) point at `dist`._ **`exports` map main entry resolves to TypeScript source, not compiled JS** ‚Äî `package.json:15-21` ‚Äî The `"."` export's `"default"` condition points to `"./src/index.ts"`. The `"./schema"` export has the same problem (`"./src/types/payload-augment.ts"`). In a published `"type":"module"` package, the `default` condition must resolve to a `.js` file; bundlers and Node will fail or silently load raw TypeScript. The `bin` entry correctly targets `dist/`, making the package internally inconsistent. **Fix:** Change all non-`types` conditions to their `dist/` equivalents (e.g. `"./dist/index.js"` for `"."`, `"./dist/types/payload-augment.js"` for `"./schema"`), and add a `"import"` condition alongside `"default"` where needed.

### üüÝ High

- [x] **`fontOverride` shallow-merge silently replaces nested objects** ‚Äî _Fixed: `upload`/`access` deep-merged and `hooks` merged via core's `mergeHooks`. `fields` still replaces (documented)._ ‚Äî `src/plugin.ts:25` ‚Äî `{ ...Font, ...fontOverride }` means that passing `fontOverride: { upload: { staticDir: './my-dir' } }` replaces the entire `upload` object, discarding the `mimeTypes` whitelist defined in the base collection. Same applies to `access` and `hooks` keys. The JSDoc comment acknowledges this but the code does not defend against it. **Fix:** Deep-merge `upload`, `access`, and `hooks` sub-keys explicitly (e.g. `upload: { ...Font.upload, ...fontOverride.upload }`, and for `hooks` spread each hook phase array). A utility like the existing `mergeHooks` in `@pro-laico/core` can handle the hooks case.

- [x] **`global` override has the same shallow-merge problem** ‚Äî _Fixed: `fields` appended to the base slots, `access` deep-merged._ ‚Äî `src/plugin.ts:30` ‚Äî `{ ...FontSet, ...global }` replaces `fields` and `access` wholesale if the consumer provides them. Providing `global: { fields: [...extraFields] }` discards the four font-slot upload fields entirely. **Fix:** For `fields`, append rather than replace: `fields: [...FontSet.fields, ...(global.fields ?? [])]`. For `access`, deep-merge per key.

- [ ] _Intentional: same source-only workspace pattern as the `exports` Critical above; only matters for a hypothetical external npm consumer._ **`package.json` `main` and `types` point to TypeScript source** ‚Äî `package.json:10-11` ‚Äî Both `"main": "./src/index.ts"` and `"types": "./src/index.ts"` are wrong for a published package. `main` must point to a JS file; `types` to a `.d.ts` file. This works in the monorepo only because other packages resolve via TypeScript project references, but it breaks any external consumer. **Fix:** Set `"main": "./dist/index.js"` and `"types": "./dist/index.d.ts"`.

- [x] **`readdirSync`/`unlinkSync` loop doesn't guard against subdirectories** ‚Äî _Fixed: entries are `statSync`-filtered to files before unlinking._ ‚Äî `src/scripts/downloadFonts.ts:117` ‚Äî `fs.unlinkSync(path.join(FONT_FILES_DIR, file))` throws `EISDIR` if any subdirectory exists inside `FONT_FILES_DIR`. **Fix:** Filter entries with `fs.statSync(filePath).isFile()` before unlinking, or use `fs.rmSync(FONT_FILES_DIR, { recursive: true, force: true })` followed by `fs.mkdirSync(FONT_FILES_DIR, { recursive: true })`.

### üü° Medium

- [x] **`authd` access allows any authenticated user to manage fonts** ‚Äî _Resolved via documentation (the package ships no role model): the shared `authd` now carries a JSDoc showing how to tighten access through `fontOverride.access` / `global.access` at registration time, rather than hard-coding a project-specific role check._ ‚Äî `src/collections/font.ts:3,7` ‚Äî `create`, `delete`, and `update` are granted to any user with a valid session. In most projects fonts are admin-only assets. **Fix:** Add a role check appropriate to the project (e.g. `req.user?.roles?.includes('admin')`) or document explicitly that callers are expected to override via `fontOverride: { access: { ... } }`.

- [x] **Duplicate `authd` definition across two files** ‚Äî _Fixed: extracted to `src/access/authd.ts`; both `font.ts` and `fontSet.ts` import it._ ‚Äî `src/collections/font.ts:3` and `src/globals/fontSet.ts:5` ‚Äî Identical `const authd: Access = ({ req }) => Boolean(req.user)` copied verbatim. **Fix:** Extract to a shared `src/access/authd.ts` and import it in both files.

- [ ] _Deferred: the CLI talks to an arbitrary consumer's Payload instance, whose generated types this package can't import. Proper fix needs the consumer to supply their generated types (or a narrowing pass); tracked alongside the schema-stub note below._ **`PayloadSDK<any>` loses all type safety** ‚Äî `src/scripts/downloadFonts.ts:70` ‚Äî The generic argument controls the shape of collection documents returned from `sdk.find` and `sdk.findGlobal`. Passing `any` means every response is untyped, leading to the cast-heavy code below (lines 136, 182, 197). **Fix:** Provide the concrete generated types from the consumer project, or at minimum use `unknown` and narrow explicitly, rather than `any`.

- [ ] _Deferred with `PayloadSDK<any>` above: because the SDK is generic-`any` here, `sdk.login`'s return is untyped, so the cast is what surfaces `token`. Removing it cleanly depends on the same typed-SDK fix. Login has already succeeded at this point, so `token` is present at runtime._ **Unsafe cast `(user as { token: string }).token`** ‚Äî `src/scripts/downloadFonts.ts:182` ‚Äî The `sdk.login` return type already includes `token` per the PayloadSDK type definitions. The cast hides a potential type mismatch and would silently produce `undefined` if the SDK changes. **Fix:** Remove the cast and use the typed return value directly; if the property truly isn't typed, file a bug against `@payloadcms/sdk` and use a type guard instead.

- [ ] _False positive: the template renders `${declarations.length === 0 ? '//eslint-disable-next-line' : ''}` on its own line, so when there are no declarations the disable comment sits **directly** above `import localFont` (no blank line); when declarations exist the import is used, so no disable is needed. Left as-is._ **`eslint-disable-next-line` comment is separated from the import by a blank line** ‚Äî `src/scripts/downloadFonts.ts:103-104` ‚Äî The generated file writes the disable comment, then a blank line, then `import localFont from 'next/font/local'`. `eslint-disable-next-line` must be on the line immediately before the offending statement; with an intervening blank line it disables nothing, and linting the generated file will still report an `unused import` error. **Fix:** Remove the blank line between the disable comment and the import in the template string, or switch to an `/* eslint-disable */` block comment wrapping both lines.

- [x] **CSS variable names are hardcoded and not configurable** ‚Äî _Fixed: added `cssVariablePrefix` option (env `ATOMIC_FONTS_CSS_VAR_PREFIX`, default `--font-set`); the slot name is appended capitalised so the emitted vars are unchanged by default._ ‚Äî `src/scripts/downloadFonts.ts:86-89` ‚Äî `--font-setSans`, `--font-setSerif`, `--font-setMono`, `--font-setDisplay` are embedded directly in the generated `localFont()` calls. Consumers who need different variable names (e.g. to integrate with a third-party design system) have no way to override them. **Fix:** Add a `cssVariablePrefix?: string` option to `RunDownloadFontsOptions` (defaulting to `'--font-set'`) and interpolate it at line 86-89.

- [ ] _Deferred (publish-only): changing `moduleResolution` to `node16`/`nodenext` for just this package risks diverging from the monorepo's bundler-resolution baseline and surfacing extensionless-import errors across shared types. Only matters for external ESM consumers of the built CLI; revisit if/when the package is published._ **`tsconfig.json` uses `moduleResolution: "bundler"` but dist is compiled by `tsc`** ‚Äî `tsconfig.build.json:1` (inherits base) ‚Äî `bundler` resolution is designed for bundler-processed outputs and permits extensionless imports. `tsc` alone does not transform these imports, so the compiled `dist/` files may contain bare specifiers that Node's ESM resolver rejects at runtime. **Fix:** Set `"moduleResolution": "node16"` (or `"nodenext"`) in `tsconfig.build.json`'s `compilerOptions` to match what Node will do at runtime, or switch to a bundler step (`tsup`, `rollup`) that handles the resolution.

### üü¢ Low / nice-to-have

- [x] **`fontUploadFields` upload fields have no `filterOptions`** ‚Äî _Fixed: each slot now sets `filterOptions: { family: { equals: '<slot>' } }`, pre-filtering the picker to matching-family fonts._ ‚Äî `src/fields/font.ts:15-22` ‚Äî Any document from the `font` collection can be selected regardless of its `family` field. A `sans` slot can be assigned a font whose `family` is `mono`. **Fix:** Add `filterOptions: { family: { equals: '<slot>' } }` to each upload field so the selector is pre-filtered to the matching family.

- [x] **`font.ts` collection has no `timestamps: true`** ‚Äî _Fixed: added `timestamps: true`._ ‚Äî `src/collections/font.ts` ‚Äî Payload does enable timestamps by default, but explicit `timestamps: true` signals intent and prevents accidental suppression if someone copies the config. **Fix:** Add `timestamps: true` explicitly.

- [ ] _Intentional fallback (documented across `@pro-laico/*`): the stub keeps the package compiling without the consumer's generated types; augmenting core's `PayloadAugment` removes the casts. Pairs with the deferred `PayloadSDK<any>` item above._ **`Font` / `FontSet` schema stubs fall back to `Record<string, any>`** ‚Äî `src/types/payload-augment.ts:6-7` ‚Äî When `PayloadAugment` is not augmented by the consumer, `Get<'Font', DefaultRecord>` resolves to `Record<string, any>`. This is intentional for compilation, but means the `font.filename` and `font.url` accesses in `downloadFonts.ts:137` are unverified at the call site, requiring the explicit cast at line 136. Document in the README that augmenting `@pro-laico/core`'s `PayloadAugment` interface with generated types removes all casts.

- [ ] _No action: `cli.ts` is **not** re-exported from `src/index.ts`, so it is unreachable via the `"."` export ‚Äî only via `bin`, which is correct. The shebang/`process.exit` are appropriate there. Left as-is._ **`src/scripts/cli.ts` is included in the `"."` export but is a Node script with side effects** ‚Äî The CLI's `#!/usr/bin/env node` shebang and `process.exit(1)` make it unsafe to import in any other context. It should not be reachable via the main export. It is currently only reachable via `bin`, which is correct, but ensuring it is explicitly excluded from any bundler entry is a worthwhile guard.

- [x] **README "Starter template" example uses `tsx` on a monorepo internal path** ‚Äî _Fixed: the bin is now labelled the canonical form and the `tsx`/`src` invocation is relabelled "In-repo development only" with a note that it isn't for published consumers._ ‚Äî `README.md:64` ‚Äî `pnpm exec tsx node_modules/@pro-laico/fonts/src/scripts/cli.ts` will only work if the consumer has `tsx` installed and the monorepo resolves `src/` correctly. The published bin (`atomic-fonts-download`) or the programmatic `runDownloadFonts` import should be the canonical recommendation. **Fix:** Move the `tsx`-based example to a "development" callout and lead with the bin or programmatic form.

## Notes

- ~~The `"./src"` export path alias in `package.json`~~ ‚Äî **Removed.** It was a redundant duplicate of `.` with no consumers; the remaining `.`/`./schema` source-resolving exports are the intentional workspace pattern.
- No hooks are added by the plugin itself, so hook-loop and transaction-threading concerns are N/A for `plugin.ts`. They would become relevant if `fontOverride` hooks are supplied by the consumer.
- `access.read` on the `Font` collection is `authd` (authenticated users only), which is appropriate for font binaries that may be licensed. The download script correctly sends a JWT for same-origin URLs, so the auth gate does not block the CLI.
- The `global` option accepting `boolean | Partial<GlobalConfig>` is ergonomic; the `typeof global === 'object'` guard at `plugin.ts:30` correctly handles both forms.

# @pro-laico/site — Payload Plugin Audit

## Summary
`sitePlugin` is a clean curried plugin that correctly appends its Pages/Header/Footer collections and Settings/SiteMetaData globals without clobbering incoming config. The collection/global configs themselves are mostly solid (correct `authenticatedOrPublished` for Pages, atomicHook + delete-revalidation wired, drafts/schedulePublish, live preview). The most serious issues are the Header/Footer frontend components declaring `'use server'` (which turns React Server Components into Server Actions) and the `href` field — the primary page-lookup key — lacking `unique`/`index`. The README also documents exports (`baseStorage`, storage globals) that no longer exist in this package.

## Findings

> **Status:** Critical + both Highs fixed — Header/Footer `'use server'`→`server-only`, `href` indexed, and the shared `updateHrefHook` null-deref guarded in core. `href` is intentionally not `unique` (derived from the breadcrumb tree). **Now also fixed:** README `baseStorage`/storage-global drift (those moved to `@pro-laico/styles`), removal of the unused `@payloadcms/ui`/`next` peers (+devDeps), the duplicated `APFunctions` list (now defined once in the Pages collection and passed into `SettingsTab(apFunctions)`), and `useAsTitle: 'href'` → `'title'` (href stays a column). **Still open — your design call:** the `site ↔ core`/`site ↔ styles` type-only circular dep (and the related `ShortcutSet` re-stub + component self-imports), which the clean fix moves to a leaf `zap`/schema package. **Intentional/annotated:** `read: authd` on the public globals (Local API `overrideAccess` posture, same as tracking/fonts) and the no-op `build` (source-only workspace package).

### 🔴 Critical
- [x] **Header/Footer frontend components marked `'use server'`** — `src/collections/headers/component.tsx:1`, `src/collections/footers/component.tsx:1` — The files start with `'use server'`, which marks the whole module as a **Server Actions** module, not a Server Component. Both export an async component that takes a non-serializable object prop (`{ header }` / `{ footer }`) and returns JSX. Server Actions require serializable args and must not return JSX; the React/Next bundler treats these as RPC endpoints, which breaks rendering and (in strict builds) errors at compile time. **Fix:** Remove the `'use server'` directive entirely — async RSCs need no directive. (Contrast with `@pro-laico/atomic/hook/lazyAtomicHook.ts`, where `'use server'` is correct because every export there is a genuine async server action.)

### 🟠 High
- [x] **`href` field has no `unique` and no `index`** — _Fixed: added `index: true`. Left non-`unique` deliberately (hrefs derive from the breadcrumb tree; a unique constraint would reject legitimate edge cases / seed reruns)._ — `src/collections/pages/collection.ts:62-68` — `href` is the canonical page identity: `getCachedPageByHref` queries `where: { href: { equals: tid } }` (core `getPage.ts:20`), it is the `useAsTitle`, the first `defaultColumn`, and the cache key for `revalidateTag('page', href)`. Without `index: true` every page lookup is an unindexed scan; without `unique: true` nested-docs slug collisions can produce two pages with the same `href`, and `limit: 1` then returns a nondeterministic one. **Fix:** add `index: true` (and `unique: true` if your data model guarantees globally-unique hrefs) to the `href` APField.
- [x] **`updateHrefHook` dereferences `data.breadcrumbs` without a guard** — _Fixed in core `hooks/field/href.ts` (guards the array before indexing); core dist rebuilt._ — `src/collections/pages/collection.ts:66` wires `updateHrefHook`, whose body is `data?.breadcrumbs[data?.breadcrumbs?.length - 1]?.url` (core `hooks/field/href.ts:4`). The optional chain is only on `data?.breadcrumbs`; the immediate `[...]` index access throws `TypeError: Cannot read properties of undefined` whenever `data` exists but `breadcrumbs` is absent — e.g. a Pages write where the nested-docs plugin is not installed, or any create before breadcrumbs are computed. **Fix:** guard the array first, e.g. `const bc = data?.breadcrumbs; if (!bc?.length) return; ...` (change lives in core, but it is load-bearing for this package's Pages collection).

### 🟡 Medium
- [x] **README documents exports/files that no longer exist** — _Fixed: removed the `baseStorage('draft')`/`('published')` bullets, the `globals/storage.ts` table row, and `baseStorage` from the exports example; added a note that the CSS storage globals now ship from `@pro-laico/styles`; reworded the intro + "Why it exists" accordingly._ — `README.md:14,43,51-57,72` — It describes `baseStorage('draft')`/`baseStorage('published')` globals, a `globals/storage.ts` file, and lists `baseStorage` in the "Notable exports" block. `plugin.ts:13-18` states those storage globals moved to `@pro-laico/styles`, `src/globals/` contains only `settings.ts`/`siteMetaData.ts`, and `src/index.ts` exports neither `baseStorage` nor a storage global. The "Notable exports" list also claims `Settings, SiteMetaData, baseStorage, SEOTab, SettingsTab` are exported alongside `Pages/Header/Footer` — `baseStorage` is not. **Fix:** delete the `baseStorage`/storage-global bullets and table row, drop `baseStorage` from the exports example, and update the "Depends on" line if storage now flows through styles.
- [x] **Unused peer dependencies `@payloadcms/ui` and `next`** — _Fixed: removed both from `peerDependencies` and `devDependencies` (re-confirmed by grep that neither is imported anywhere under `src/`); typecheck still passes._ — `package.json:46-48` — Neither is imported anywhere under `src/` (confirmed by grep; the admin component is referenced only via the string path `APFControlsPath`, and the JSX components rely on `react` only). Declaring them as peers forces consumers to install/resolve packages this code never touches. **Fix:** remove `@payloadcms/ui` and `next` from `peerDependencies` (and their `devDependencies` entries) unless a future export needs them.
- [ ] _**Design decision — the core↔siblings circular-workspace-deps item.** Type-only today so it resolves, but the clean fix (move the shared schema-stub types to a leaf `@pro-laico/zap`/dedicated-schema package both sides depend on) spans core, styles, and site. Holding for your call rather than shuffling stubs piecemeal._ **Circular package dependency via `@pro-laico/site/schema`** — `src/collections/headers/component.tsx:3`, `src/collections/footers/component.tsx:3`, `src/types/payload-augment.ts:7` import from `@pro-laico/site/schema`, while `core` (`types/cache.ts:12`, `utilities/generateMetaData.ts:2`) and `styles` (`types/payload.ts:7`, `cssProcessor.ts:2`, `ui/rowLabels/shortcut/index.tsx:4`) import the same subpath — yet `site`'s `package.json:39-43` depends on both `core` and `styles`. This is a real `site ↔ core` and `site ↔ styles` cycle (type-only today, so it resolves, but it is fragile and will break any value-level import added across the boundary). **Fix:** move the shared schema-stub types into a leaf package (e.g. `@pro-laico/zap` or a dedicated `schema` package) that both sides depend on, so the dependency graph stays acyclic.
- [x] **Duplicated APFunctions list that can silently drift** — _Fixed: the `APFunctions` array is defined once in the Pages collection (feeding the admin `APFControlsPath` clientProps) and now passed into `SettingsTab(apFunctions)`, which threads it to `generateAPFFields` — single source of truth._ — `src/collections/pages/collection.ts:23` defines `APFunctions = ['form','page','pages','actions','classes','seo','sitemap']` for the admin controls, and `src/collections/pages/tabs/settings.ts:11` hardcodes the identical literal again for `generateAPFFields(...)`. If one list changes, the admin controls and the generated storage fields fall out of sync. **Fix:** export the array once (e.g. from the collection or a shared constant) and pass it into `SettingsTab(apFunctions)`.
- [x] **`changeFrequency` select has no `defaultValue` despite "Defaults to monthly"** — _Fixed: `defaultValue: 'monthly'`._ — `src/collections/pages/tabs/SEO.ts:56-62` — the field stores `null` until the editor picks a value; the "monthly" default only exists as a code fallback in the sitemap getter (`getSitemap.ts:49,56`). The description on `SEO.ts:9` promises a default that the schema does not provide, so any other consumer reading `meta.changeFrequency` gets `null`. **Fix:** add `defaultValue: 'monthly'` to the `changeFrequency` APField.

### 🟢 Low / nice-to-have
- [x] **`useAsTitle: 'href'` yields a blank admin title for new/unsaved pages** — _Fixed: `useAsTitle: 'title'` (always populated, required); `href` stays in `defaultColumns`._ — `src/collections/pages/collection.ts:35` — `href` is computed from breadcrumbs by `updateHrefHook` and is empty until the first save, so the admin list/title shows blank for freshly created docs. Pages already has a required `title` field. **Fix:** consider `useAsTitle: 'title'` (it stays in `defaultColumns` regardless), keeping `href` as a column.
- [ ] _Deferred — same root as the circular-dep item above: where `ShortcutSet` should live (it's owned by styles) is exactly what the "move shared stubs to a leaf package" decision resolves. Dropping it from here now would just point site at styles and tighten the cycle. `SiteMetaDatum` is site-owned and fine._ **`types/payload.ts` re-stubs non-site shapes** — `src/types/payload.ts:9-10` declares `ShortcutSet` and `SiteMetaDatum`. `ShortcutSet` is owned by `@pro-laico/styles`, not site; duplicating its stub here means two packages can disagree on the type. `SiteMetaDatum` (note the singular Payload-generated name) is site-owned and fine. **Fix:** drop `ShortcutSet` from this file and import it from styles' schema where needed.
- [ ] _Intentional posture (consistent across `@pro-laico/*`): writes are gated and SSR reads go through the cache getters' Local API default `overrideAccess: true`, so anonymous REST/GraphQL reads stay denied while public pages still get their metadata. Could be made explicit with `() => true` later, but the current behaviour is correct; left as-is to match tracking/fonts._ **`SiteMetaData` / `Settings` globals use `read: authd`** — `src/globals/siteMetaData.ts:11`, `src/globals/settings.ts:17` — these are read for public SSR (`getCachedSiteMetadata` etc.). It works only because the cache getters use the Local API default `overrideAccess: true`; the access predicate is never satisfied for an anonymous REST/GraphQL read. If anything ever fetches site metadata with `overrideAccess: false`, public pages lose their metadata. **Fix:** if site metadata is meant to be publicly readable, use a `() => true` read predicate (or `authenticatedOrPublished`) so the access intent is explicit rather than relying on overrideAccess defaults.
- [ ] _Deferred — same root as the circular-dep item: switching to the local relative type only pays off once the `Header`/`Footer` zap registry entries are populated (otherwise they still fall back to `Record<string, any>`), which is part of the shared-schema-package decision. Works today; revisit with that change._ **Frontend components self-import `@pro-laico/site/schema`** — `src/collections/headers/component.tsx:3`, `footers/component.tsx:3` — importing the package's own subpath from inside the package adds a self-reference and means `header.children` / `header.ClassName` resolve to `any` (`Get<'Header', DefaultRecord>` falls back to `Record<string, any>` when the zap registry has no `Header`). Works, but loses type safety on the rendered fields. **Fix:** import the local relative type (`../../types/payload`) and ensure the zap `Header`/`Footer` registry entries are populated so `children`/`ClassName` are typed.
- [ ] _Intentional: source-only workspace package (`main`/`types` → `./src/index.ts`, consumed via `transpilePackages`), the standard `@pro-laico/*` pattern. No emit needed unless publishing to npm for non-bundler consumers._ **`build` script is a no-op** — `package.json:35` (`node -e "process.exit(0)"`). The package publishes raw `src/` (`main`/`types` point at `./src/index.ts`) and ships `tsconfig.build.json`, which is never invoked. Intentional for a source-only workspace package, but if this is ever published to npm for non-bundler consumers, there is no emitted JS/d.ts. **Fix:** none if source-only is the contract; otherwise wire `tsc -p tsconfig.build.json` into `build` and point `exports` at `dist`.

## Notes
- Plugin architecture is correct: `sitePlugin` is curried `(options) => (config) => Config`, honors `enabled === false`, and **appends** (`[...(config.collections ?? []), ...]`) rather than replacing. It deliberately does no `onInit` composition, no nested-docs wiring, and no overrides surface — those are explicitly delegated to the template's composition layer (`plugin.ts:13-18`), which is a reasonable design choice for this package. There is no per-collection toggle or `overrides` option; acceptable given the "fixed site shape" intent, but worth documenting as a non-goal.
- Access on Pages is correct and safe: `read: authenticatedOrPublished` (`access.ts:9`) plus the getters' `overrideAccess: draft` and `live: true` / `_status: published` where-clauses (core `getPage.ts`, `getPages.ts`, `getSitemap.ts`) prevent drafts/unpublished pages leaking to anonymous frontend reads.
- Header/Footer "single active" invariant is enforced at write time by `atomicHook` → `unsetActive` (atomic `createAtomicHook.ts:83`) and the active flag is `index: true` (`core apf/fields/active.ts`), so the `where: { active: true }, limit: 1` getters are fine; no schema-level uniqueness needed. `read: authd` on Header/Footer is intentional (no public API read) and does not break the frontend because getters use the overrideAccess default.
- Revalidation is wired correctly without an explicit `afterChange` hook: `atomicHook` (beforeChange) performs `revalidateTag` for pages/page/sitemap/classes/etc. internally and is context-guarded (`isSeed`, `atomicHookRan`, per-apf `runAPF` id guards), and `revalidateCacheOnDelete` (afterDelete) handles header/footer/pages delete invalidation. Settings/SiteMetaData globals use `revalidateCacheGlobal`, whose switch covers both slugs.
- `RequestContext` augmentation (`types/payload-augment.ts`) correctly matches the keys `atomicHook` writes via `context[slug] = data` for `pages`/`header`/`footer`.

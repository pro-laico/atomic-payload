# @pro-laico/tracking — Payload Plugin Audit

## Summary

The plugin is structurally sound: it follows the `(options)=>(config)=>Config` curried shape, spreads the incoming config, and appends rather than replaces collections/globals. The main risk areas are (1) `posthog.init` being called on every `pathname` change instead of once, which re-initialises the SDK repeatedly; (2) the `Tracking` global is fully locked behind `authd` access, which is correct for writes but means the client-side provider must receive the global value server-side (not a problem if the parent fetches it, but the README example omits the `tracking` prop entirely); (3) the `./provider` subpath export is missing a `react-server` condition pointing to a safe stub, so Next.js server builds may still resolve the client bundle; and (4) `postHogPublicKey` is stored in a Payload document with no field-level access restriction, meaning any authenticated editor can read it — which is acceptable for a *public* key but is undocumented. Minor issues include the unnamed `group` field in `postHogTabField`, absent `index` on frequently queried fields, and a stale README usage example.

> **Status:** Criticals resolved — `posthog.init` re-init fixed (both items, via `__loaded` guard); the group-`name` "critical" is a false positive (unnamed groups hoist to parent — annotated); `read: authd` and `./schema`→`.ts` are intentional (annotated); `./provider` react-server stub deferred (low benefit). The stale `dist/` (deleted-`ap-utils` import) was clean-rebuilt. **Now also fixed:** shared `authd` (`src/access/authenticated.ts`), conditional-`required` for `postHogPublicKey`/`postHogHost`/`googleTagManagerId` (enforced only when the provider is enabled), `postHogTabField(): TabAsField` return annotation, empty-URL `.filter(Boolean)` + a posthog-js version comment, the README prop-threading example, and removal of the redundant `./src` alias. The `Tracking` global `beforeChange`→`afterChange` revalidation is now done (new core `revalidateCacheGlobalAfterChange`, also applied to `siteMetaData`/`settings`). **Deferred:** the `posthogProperty` slug rename (breaking migration) and `timestamps: false` (schema migration; minor). `main`/`types`→`src` is the intentional workspace pattern.

## Findings

### 🔴 Critical

- [x] **`posthog.init` called on every pathname change** — _Fixed: a `posthog.__loaded` guard makes init run at most once; pageviews are left to PostHog's automatic capture (`defaults`) to avoid double-counting. Component dist rebuilt._ — `src/components/frontend/PostHogProvider.tsx:10–37` — `posthog.init` is placed inside a `useEffect` that lists `pathname` as a dependency (line 38). Every client-side navigation therefore re-initialises the PostHog SDK. PostHog is a singleton; calling `init` repeatedly produces duplicate event capture and can corrupt session state. **Fix:** Split into two effects — one with `[]` deps that calls `posthog.init` once, and a separate `useEffect([pathname])` that calls `posthog.capture('$pageview')` (or rely on PostHog's automatic page-view capture). Remove `pathname` from the init effect's dependency array.

### 🟠 High

- [ ] **`./provider` export map missing `react-server` condition** — _Deferred: real optimization, but `TrackingProvider` is `'use client'` so the RSC boundary already prevents server execution of `posthog-js`. Implementing needs a server-stub file + `react-server` condition + rebuild; low marginal benefit._ — `package.json:26–29` — The `./provider` subpath only declares `types` and `import` conditions. When Next.js resolves the package under the `react-server` module condition (e.g. in Server Components or during `payload generate:importmap`), bundlers fall through to the `import` condition and load the full client bundle, which imports `posthog-js` and `@next/third-parties`. This is the exact problem the subpath was created to prevent. **Fix:** Add a `"react-server": "./dist/components/frontend/server-stub.js"` entry (before `import`) that exports `null`-returning stubs or simply `{}`, and create the corresponding stub file.

- [ ] **`Tracking` global `read` access locked to authenticated users but provider receives data server-side** — _Intentional, same posture as `@pro-laico/site`/`fonts`: the cache getter reads via the Local API default `overrideAccess: true`, so SSR works while anonymous REST/GraphQL reads stay denied. Left as-is; could be made explicit later._ — `src/globals/tracking.ts:13` — `access: { read: authd }` means any unauthenticated Local API call (e.g. `payload.findGlobal({ slug: 'tracking', overrideAccess: false })`) will be denied. The README example (line 38) shows `<TrackingProvider>` with *no* `tracking` prop, implying callers must fetch the global themselves. If they call the Local API without `overrideAccess: true` from a server component, they get a 403. **Fix:** Either document that callers must use `overrideAccess: true` in the fetch (and explain why that is safe), or split `read` into two tiers — `read: () => true` for the server fetch path and a separate field-level access guard on any sensitive fields (there are none currently). At minimum, update the README example to show the fetch + prop threading.

- [x] **`postHogTabField` `group` field has no `name`** — _NOT a bug (analysis): an UNNAMED Payload group is presentational and hoists its subfields to the parent, so `enableAutoCapture`/`disableSessionRecording`/… persist at the global root — exactly where `PostHogProvider` reads them (`tracking.enableAutoCapture`). Adding a `name` would nest them and break the provider. Left as-is intentionally._ — `src/globals/postHogTab.ts:12` — The inner `type: 'group'` at line 12 (`label: 'Feature Toggles'`) is missing a `name` property. Payload requires a `name` on named group fields to persist data; without it the field is a layout-only UI row and Payload will not store `enableAutoCapture`, `disableSessionRecording`, `disableSurveys`, or `capturePerformance` under any namespace. These values will appear in the admin UI but will not be addressable via the API. **Fix:** Add `name: 'postHogFeatureToggles'` (or similar) to the group, then update `PostHogProvider.tsx` to read from `tracking.postHogFeatureToggles.enableAutoCapture` etc.

### 🟡 Medium

- [x] **`posthog.init` called even when PostHog is not initialised yet — no `posthog.__loaded` guard** — _Fixed by the same `__loaded` guard above._ — `src/components/frontend/PostHogProvider.tsx:24` — There is no check for `posthog._isInitialised` or `posthog.__loaded` before calling `init`. If the component renders twice in React Strict Mode (development), `init` is invoked twice in rapid succession before the first call completes, which can trigger PostHog's internal duplicate-init warning. **Fix:** Guard with `if (!posthog.__loaded) { posthog.init(...) }`.

- [ ] **`./schema` export points to `.ts` source, not compiled output** — _Intentional workspace-only source consumption (consistent across all `@pro-laico/*` packages)._ — `package.json:17–20` — The `./schema` entry exposes `./src/types/payload-augment.ts` directly as both `types` and `default`. At runtime this is a raw TypeScript file. Bundlers that honour the `default` condition will fail unless the consuming app transpiles `node_modules`. **Fix:** Add a built output: compile `src/types/payload-augment.ts` (it is already included in `tsconfig.build.json`) and point `default` to `./dist/types/payload-augment.js`; keep `types` pointing to `./dist/types/payload-augment.d.ts`.

- [ ] _Intentional: source-only workspace consumption (`.` resolves to `src`), the standard `@pro-laico/*` pattern; only `./provider` points at `dist` because it's a genuine client bundle. Matters only for a hypothetical external npm consumer._ **`main` and `types` point to source `.ts`, not `dist`** — `package.json:10–11` — `"main": "./src/index.ts"` and `"types": "./src/index.ts"` are only appropriate if every consumer uses a bundler that transpiles `node_modules` (e.g. Next.js with `transpilePackages`). A plain Node.js consumer or Jest runner without transform will crash. The `exports` map overrides `main` in modern Node, but `types` is also used by TypeScript path resolution in older `moduleResolution` modes. **Fix:** Point `main` to `./dist/index.js` and `types` to `./dist/index.d.ts`; keep `src` paths only in the `exports` map for bundler-transpiled consumers if desired.

- [x] **`Tracking` global uses `beforeChange` hook for cache revalidation, not `afterChange`** — _Fixed: added `revalidateCacheGlobalAfterChange` (`GlobalAfterChangeHook`) to `@pro-laico/core` and switched the `Tracking` global to `hooks: { afterChange: [...] }`, so the cache is busted only after the write commits. The same swap was applied to `siteMetaData` and `settings` in `@pro-laico/site`._ — `src/globals/tracking.ts:25` — `revalidateCacheGlobal` (`revalidate.ts:9`) fires in the `beforeChange` phase. If the DB write subsequently fails, the cache has already been invalidated and the stale or empty data will be served until the next successful save. **Fix:** Move the hook to `afterChange` so revalidation only happens once the document is durably persisted.

- [x] **`postHogTabField` `postHogPublicKey` marked `required: true` but field is inside a conditionally shown tab** — _Fixed: replaced the hard `required: true` on `postHogPublicKey` and `postHogHost` with a `requiredWhenEnabled('postHogEnabled', …)` validate, so the global saves fine with PostHog off and still enforces the keys when it's on._ — `src/globals/postHogTab.ts:9` — `postHogPublicKey` is `required: true`. The tab is conditionally shown when `postHogEnabled` is true, but Payload validates `required` regardless of `admin.condition`. If a user saves the Tracking global with PostHog disabled, they will receive a validation error because `postHogPublicKey` is empty. **Fix:** Either set `required: false` and handle the missing key at runtime (already guarded at `PostHogProvider.tsx:21`), or add a `validate` function that only enforces the requirement when `postHogEnabled` is `true`. Same applies to `postHogHost`.

- [x] **`googleTagManagerId` not marked `required`** — _Fixed: added the same `requiredWhenEnabled('googleTagManagerEnabled', …)` validate, enforcing the ID only when GTM is enabled._ — `src/globals/gtmTab.ts:9` — The GTM ID field has no `required: true`. GTM cannot function without an ID; the `GoogleTagManagerProvider` silently skips rendering if the ID is absent (line 9), but there is no validation warning in the admin. **Fix:** Add `required: true` to `googleTagManagerId` (and apply the same conditional-required approach as the PostHog fix above — only enforce when `googleTagManagerEnabled` is `true`).

### 🟢 Low / nice-to-have

- [x] **`authd` access function duplicated in two files** — _Fixed: extracted to `src/access/authenticated.ts`; both files import it._ — `src/collections/posthogProperty.ts:3` and `src/globals/tracking.ts:7` — Identical `const authd: Access = ({ req }) => Boolean(req.user)` defined separately. **Fix:** Extract to a shared `src/access/authenticated.ts` and import in both files.

- [x] **`postHogTabField` return type is inferred `any` via `storageField`** — _Fixed: annotated `postHogTabField(): TabAsField`, so a future shape mismatch (e.g. a group losing its `name`) surfaces at compile time._ — `src/globals/postHogTab.ts:3` — The function signature is `() => { ... }` with the return type inferred from `storageField: TabAsField`. The inner unnamed group (see High finding above) may silently typecheck even though it lacks `name`. **Fix:** Annotate the function return as `: TabAsField` explicitly to surface future type drift.

- [ ] _Deferred: renaming the slug is a breaking migration for any existing install (changes the REST path and the stored collection). Left for a dedicated migration; not a safe in-place cleanup._ **`posthogProperty` slug uses camelCase** — `src/collections/posthogProperty.ts:14` — Payload collection slugs conventionally use kebab-case (e.g. `posthog-property`) to match REST URL segments. `posthogProperty` produces `/api/posthogProperty` which is non-standard. Changing this is a **breaking migration** if documents already exist. **Fix:** Use `posthog-property` for new installs; if migrating, provide a migration script and document the slug change.

- [ ] _Deferred: setting `timestamps: false` drops two existing columns — a schema migration for any live install — for a marginal at-scale gain. Left as-is._ **`postHogProperty` collection has no `timestamps` override** — `src/collections/posthogProperty.ts:14` — Payload adds `createdAt`/`updatedAt` by default; for a pure reference collection (no time-sensitive data) this wastes two indexed columns. If temporal tracking is not needed, consider `timestamps: false`. Low priority — only matters at scale.

- [x] **README Quick Start example omits `tracking` prop** — _Fixed: the frontend example now fetches the `Tracking` global server-side with `getPayload(...).findGlobal({ slug: 'tracking' })` and threads it into `<TrackingProvider tracking={tracking}>`, with a note on the authenticated-read / `overrideAccess` default._ — `README.md:38` — `<TrackingProvider>` in the Quick Start does not pass `tracking`, so all providers short-circuit at line 18 (`if (!tracking) return <>{children}</>`). The example is non-functional as written. **Fix:** Update the example to show fetching the Tracking global server-side and threading it as a prop.

- [x] **`postHogAutoCaptureSettings` URL arrays allow empty strings** — _Fixed: `.filter(Boolean)` now drops empty/falsy URLs before they reach PostHog, so an unfilled row no longer becomes a match-everything pattern._ — `src/components/frontend/PostHogProvider.tsx:29–30` — `item.url ?? ''` falls back to an empty string `''`, which PostHog will treat as a match-everything pattern. **Fix:** Filter out falsy values: `.filter(Boolean)` before passing to PostHog.

- [x] **`url_allowlist` / `url_ignorelist` are PostHog v1 option names** — _Fixed: added an inline comment noting these are posthog-js autocapture options tied to the `defaults: '2025-05-24'` snapshot, pointing maintainers at the changelog for version drift._ — `src/components/frontend/PostHogProvider.tsx:29–30` — PostHog JS >=1.360 uses `url_allowlist` as a `RegExp | string` array within `autocapture`, but the exact key name may vary by version. Pinning a `defaults: '2025-05-24'` snapshot (line 26) partially mitigates drift, but documenting the posthog-js version sensitivity here would help maintainers. **Fix:** Add a comment referencing the PostHog changelog version that introduced these options.

## Notes

- The `revalidateCacheGlobal` hook imported from `@pro-laico/core` hardcodes a `switch` on the global slug (`tracking`). This is correct and will revalidate the `'tracking'` Next.js cache tag. However, the hook lives in `core` and must be kept in sync if the slug ever changes — consider documenting this coupling.
- ~~The `./src` export alias (`package.json:21–24`)~~ — **Removed** (redundant duplicate of `.`, no consumers).
- `GoogleTagManagerProvider` is marked `'use client'` but `<GoogleTagManager>` from `@next/third-parties/google` is itself a Server Component that injects a `<script>` tag. Wrapping it in a client component works but prevents the script injection from happening during SSR, which delays GTM load. Consider refactoring to a Server Component that conditionally renders `<GoogleTagManager>` directly in the layout.
- `VercelProvider` wraps `<Analytics />` in `'use client'` unnecessarily — `@vercel/analytics/next` exports a Server Component. Same concern as GTM above.
- There is no `onInit` composition in `plugin.ts`. This is fine as the plugin currently has no init-time side effects, but the absence is worth noting if side effects are added later.

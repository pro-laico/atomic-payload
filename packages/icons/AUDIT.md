# @pro-laico/icons ‚Äî Payload Plugin Audit

## Summary

The plugin architecture is sound: it correctly curries `(options) => (config) => Config`, preserves the incoming config, and handles the `enabled` flag. The most serious issues are an SVG XSS vector (`dangerouslySetInnerHTML` inlines unstripped payloads from `svgString` that is never sanitized against `<script>` or event-handler injection), `'use server'` on a plain hook module that is neither an async Server Action nor a Next.js route, and a `console.*` logger used throughout instead of `req.payload.logger`. The packaging has a split exports map (some subpaths point to `src/`, others to `dist/`), and blocks are excluded from the build output while their `package.json` exports point to `src/` ‚Äî this works only because consumers are expected to use the monorepo bundler, but it is fragile for published builds.

> **Status:** üî¥ Criticals fixed (XSS sanitization + `'use server'` removed). Also fixed: `console.*`‚Üílogger, the `'use server'` directives on `Icon.tsx`/`iconChild`, `aria-hidden` default, and the transform/clip-path early-return now stores a sanitized `svgString`. Access-control and packaging items are intentional for workspace consumption (see Notes). **Now also fixed:** `beforeChange`‚Üí`afterChange` revalidation on **both** the `icon` and `iconSet` collections (iconSet wasn't separately flagged but had the identical staleness bug), the `extractSvgProps`/`extractSvgContent` regexes (namespaced/hyphenated attrs, single quotes, lazy nested-`<svg>` match), and doc comments on the `iconChild` icon-field trade-off and the `maxPerDoc: 50` retention limit. Dist rebuilt.

## Findings

### üî¥ Critical

- [x] **SVG XSS: `<script>` and event-handler injection not stripped** ‚Äî `src/hooks/formatSVG.ts:14-17` ‚Äî The hook bails out early for `transform=` and `clip-path=` but does **not** strip `<script>` elements, `on*=` event attributes, or `javascript:` hrefs before passing the SVG through SVGO. SVGO's `preset-default` does **not** guarantee removal of `<script>` tags or inline event handlers in all configurations, and the stored `svgString` is later injected verbatim via `dangerouslySetInnerHTML` in `Icon.tsx:53`, `iconChild/component.tsx:12,18`. An uploaded SVG with `<script>alert(1)</script>` or `<path onload="...">` will be stored and then executed in the browser for any visitor. **Fix:** add an explicit SVGO plugin config entry `{ name: 'removeScripts' }` (removes `<script>`) and `{ name: 'removeEventHandlers' }` (removes `on*` attrs) to the plugins array, or run DOMPurify / a dedicated SVG sanitizer over `svgStr` before writing it to `icon.svgString`. These must run even when the early-return paths fire.

- [x] **`'use server'` on a non-Action module** ‚Äî `src/hooks/formatSVG.ts:1` ‚Äî The directive `'use server'` at the top of a module marks **every export** as a callable Server Action endpoint accessible via network POST. `formatSVGHook` and `formatSvg` are Payload hooks, not Server Actions; labelling them this way exposes them as callable network endpoints and may conflict with how bundlers handle the module. **Fix:** Remove the `'use server'` directive from `formatSVG.ts`. No directive is needed for a plain server-side module; use `import 'server-only'` instead if you want to guard against accidental client-bundle inclusion.

### üüÝ High

- [x] **`console.*` used instead of `payload.logger`** ‚Äî `src/hooks/formatSVG.ts:15,108,131,135,146` and `src/components/admin/iconSelect.tsx:14` ‚Äî Payload's structured logger (`req.payload.logger.warn(...)` / `.info(...)` / `.error(...)`) is the correct surface; `console.*` bypasses any log-level filtering, redaction, or JSON formatting the host project configures. **Fix:** Thread `req.payload.logger` from the hook's `req` argument and replace every `console.*` call. `iconSelect.tsx` is a server component ‚Äî pass `payload` from context or accept it as a prop; alternatively use `payload.logger` imported from the Payload instance.

- [ ] _Intentional (see Notes): the frontend inlines icons via the cached `svgString` server-side, never the public `/api/icon/file` URL, so guests never need file read access. `read: authd` matches the "assets aren't public" posture (same as `@pro-laico/fonts`). Left as-is._ **Access control: `Icon` collection is fully locked to authenticated users only, no public read** ‚Äî `src/collections/icon.ts:98` ‚Äî SVG files served by Payload's file route are read-access-controlled via the `read` handler; `authd` blocks **all** unauthenticated reads, including the raw file URL. If the front-end renders icons by their upload URL (not via `svgString`), guest visitors get 401 errors. **Fix:** Expose `read: () => true` (or a more granular `read: ({ req }) => Boolean(req.user) || true`) if icons are intended to be publicly viewable assets, as is typical for static media. Keep `create/update/delete` authenticated.

- [ ] _Intentional: `.`/`./Icon`/block subpaths ‚Üí `src` is the standard `@pro-laico/*` `transpilePackages` consumption pattern; only the admin components (`./admin/*`) need `dist`, and they correctly point there. Matters only for a hypothetical non-bundler npm consumer._ **Exports map inconsistency: critical subpaths point to `src/` without a build step** ‚Äî `package.json:12-20` ‚Äî The `.` and `./src` exports resolve to `src/index.ts` (TypeScript source), and `./Icon`, `./blocks/iconChild`, `./blocks/svgChild`, and their component subpaths all point to `src/**`. An external consumer that imports `@pro-laico/icons` or `@pro-laico/icons/Icon` from a plain Node.js context or an isolated build will get raw `.ts`/`.tsx` files, not compiled JavaScript. Admin subpaths (`./admin/iconRowLabel`, `./admin/iconSelect`) correctly point to `dist/`. **Fix:** Either build all subpaths to `dist/` and point every export to `dist/`, or document explicitly that this package requires a bundler (e.g. Next.js with `transpilePackages`) and add `"bundler-only": true` to `publishConfig` to warn consumers.

- [ ] _Intentional: the block subpaths resolve to `src` (workspace consumption), so excluding `src/blocks/**` from the dist build is consistent ‚Äî same decision as images/mux-video. (The block `'use server'` directives were removed in the Critical fix.)_ **Blocks excluded from build output** ‚Äî `tsconfig.build.json:12` ‚Äî `"exclude": ["dist", "node_modules", "src/blocks/**"]` means `src/blocks/iconChild/block.ts`, `src/blocks/iconChild/component.tsx`, `src/blocks/svgChild/block.ts`, and `src/blocks/svgChild/component.tsx` are never compiled to `dist/`. The corresponding `package.json` exports point directly to `src/`. The block components use `'use server'` (iconChild) ‚Äî a TypeScript/JSX file being loaded as ESM at runtime without compilation will fail in any non-bundler environment. **Fix:** Remove `src/blocks/**` from `tsconfig.build.json`'s `exclude` list, compile them, and point the blocks exports to `dist/`.

### üü° Medium

- [x] **`extractSvgProps` attribute regex is too narrow** ‚Äî _Fixed: key pattern broadened to `[\w:.-]+` (captures `xmlns:xlink`, `xml:space`, `fill-rule`, `stroke-width`, ‚Ä¶) and the value pattern now accepts both `"‚Ä¶"` and `'‚Ä¶'`._ ‚Äî `src/utilities/extractSVG.ts:7` ‚Äî The regex `(\w+)="([^"]*)"` only captures `word=` attribute names and double-quoted values. SVG attributes like `xml:space`, `xmlns:xlink`, `fill-rule`, `stroke-width`, and single-quoted values (valid XML) are silently dropped. When `extractSvgProps` is spread onto the `<svg>` JSX element, missing `xmlns` will cause React to emit a warning; missing `viewBox` will break sizing. **Fix:** Broaden the key pattern to `[\w:.-]+` and add a branch for single-quoted values: `(?:\"([^\"]*)\"|'([^']*)')`.

- [x] **`formatSvg` silently returns original icon on transforms/clip-paths** ‚Äî `src/hooks/formatSVG.ts:13-17` ‚Äî When `hasTransforms || hasClipPaths` is true the function returns early with the unoptimized icon, but neither `svgString` nor `optimized` are updated, so the stored document retains whatever was there from a previous upload (or nothing on create). The admin field `svgString` will show stale/missing data with no indication to the user that processing was skipped. **Fix:** Return `{ ...icon, svgString: svg, optimized: 'Skipped (transform/clip-path present)' }` from the early-return branch so the field always reflects the current upload.

- [ ] _Reviewed ‚Äî no code change (the finding itself concludes the condition is sound): `req.file` is the real gate, so a metadata-only update without a new file returns early; the `create || update` pair is correct because a file replace can arrive via either. Left as-is._ **`formatSVGHook` fires on every `update` even when no new file is uploaded** ‚Äî `src/hooks/formatSVG.ts:141-151` ‚Äî The guard `if (data?.filename && req.file)` is correct only at a glance: `data.filename` is the stored filename from the previous save on metadata-only updates in some Payload versions, and `req.file` being present is the correct gate. However if `data.filename` is populated from a previous upload while `req.file` is absent the function returns early cleanly; the concern is the reverse: an update that carries `req.file` but changes only metadata would still re-process. The real issue is that `operation === 'update'` with `req.file` defined means a file is being re-uploaded, which is correct ‚Äî but there is no guard against the `create` path being called when Payload internally triggers `create` during a replace-file update, which could cause double-processing. This is a low-risk edge case but the condition `operation === 'create' || operation === 'update'` combined with `req.file` check is sound. **Fix:** No code change needed, but add a comment explaining why both operations are included.

- [x] **`iconChild` block: `icon` field is type `text` with a custom admin component, not type `upload`** ‚Äî _Documented: added a comment on the field explaining the deliberate trade-off ‚Äî it stores the icon *name* (resolved against the active IconSet at render), so there's no referential integrity, and it's intentionally not `required` so a block can be saved before an icon is picked. (Left as `text` by design rather than forcing `required`/an upload relationship, which would break existing data and the IconSet-resolution model.)_ ‚Äî `src/blocks/iconChild/block.ts:37` ‚Äî The field `{ name: 'icon', type: 'text', admin: { components: { Field: { path: IconSelectPath } } } }` stores the icon name as a plain string. This is intentional (resolves via the active IconSet at render time), but it means there is no referential integrity ‚Äî if an icon is renamed or deleted the `text` value silently becomes stale. There is also no `required: true`, so a block can be saved with no icon name. **Fix:** Add `required: true` to the `icon` text field and add a validation function that checks the value exists in the active icon set, or document the non-referential-integrity trade-off explicitly.

- [x] **`iconChild/component.tsx` missing `'use client'` boundary guard ‚Äî imports `draftMode` without explicit server guard** ‚Äî `src/blocks/iconChild/component.tsx:1` ‚Äî The component has `'use server'` (see Critical #2 for the directive issue), but also critically: it imports `getCached` from `@pro-laico/core/cache/auto` which internally calls `draftMode()` from `next/headers`. If this component is ever imported on the client side (e.g. in a bundler that does not respect the directive), it will throw at runtime. **Fix:** After removing `'use server'`, add `import 'server-only'` at line 1 to produce a compile-time error on invalid client imports.

- [x] **`Icon` component does not set `aria-hidden` based on icon role** ‚Äî `src/components/frontend/Icon.tsx:53` ‚Äî The rendered `<svg>` has no default `aria-hidden="true"` or `role="img"` + `<title>` handling. Decorative icons should be `aria-hidden`; meaningful icons should have `<title>` and `role="img"`. The JSX spread allows callers to add these, but the default is incorrect for most icon uses (decorative). **Fix:** Default `aria-hidden` to `"true"` in the spread: `<svg aria-hidden="true" {...extractSvgProps(source)} {...svgProps} ...>`. Callers who want the icon to be announced can override with `aria-hidden={false}` and pass a `title` child.

- [x] **`revalidateCacheCollection` placed in `beforeChange` hook on `Icon`** ‚Äî _Fixed in the cross-cutting pass: now `afterChange: [revalidateCacheCollectionAfterChange]` (post-commit). Dist rebuilt._ ‚Äî `src/collections/icon.ts:110` ‚Äî Cache revalidation in `beforeChange` fires before the document is actually written to the database. If the database write subsequently fails (validation error, transaction rollback), the cache has already been invalidated, causing a cache-miss on the old (still-valid) data until the next genuine write. **Fix:** Move `revalidateCacheCollection` to `afterChange` so invalidation only fires when the write has succeeded.

### üü¢ Low / nice-to-have

- [x] **`'use server'` on `Icon.tsx` is redundant but harmless in Next.js App Router** ‚Äî `src/components/frontend/Icon.tsx:1` ‚Äî Async React Server Components do not require `'use server'`; that directive is for Server Actions. In Next.js App Router, a component file with no `'use client'` is already server-only. The directive here does not cause a bug (Next.js ignores it on component exports that are not called via form actions), but it is misleading and may confuse contributors. **Fix:** Remove the `'use server'` directive; add `import 'server-only'` if a hard import-boundary guard is desired.

- [x] **`extractSvgContent` regex is greedy and will mis-parse nested SVGs** ‚Äî _Fixed: switched to the lazy quantifier `([\s\S]*?)` so a nested `<svg>` no longer makes the match run to the outer document's last `</svg>` (a full parser is still preferable for deeply nested cases, noted in a comment)._ ‚Äî `src/utilities/extractSVG.ts:1` ‚Äî The pattern `<svg[^>]*>([\s\S]*)<\/svg>` uses a greedy `[\s\S]*`, which will match from the first `<svg>` to the **last** `</svg>` in the string. If an SVG contains a nested `<svg>` element (valid per spec), the extracted content will include the closing tag of the inner `</svg>`. **Fix:** Use a lazy quantifier: `<svg[^>]*>([\s\S]*?)<\/svg>` ‚Äî though for deeply nested SVGs a proper parser is preferred.

- [x] **`IconLabelPath` exported from root barrel, but the component it refers to uses `'use client'`** ‚Äî _Real issue, now fixed at the consumer: once `formatSVG.ts` swapped `'use server'`‚Üí`'server-only'` (Critical fix), the `@pro-laico/icons` barrel became a hard `server-only` surface (it exports the `Icon` collection, which transitively imports `formatSVG`). A `'use client'` component ‚Äî `@pro-laico/atomic/.../AtomicRowLabel.tsx` ‚Äî imported `{ AtomicIcon }` from that barrel, dragging `server-only` into the admin client graph and crashing the admin build. **Fix:** added a narrow `@pro-laico/icons/AtomicIcon` export and pointed `AtomicRowLabel` at it (`AtomicIcon` is a pure presentational SVG). Client components must use narrow subpaths, never the barrel. (`formatSVGHook`/`formatSvg` stay in the barrel ‚Äî only server config imports it now; no consumer imports them client-side.)_ ‚Äî `src/index.ts:13` ‚Äî `IconLabelPath` is a string constant pointing to `@pro-laico/icons/admin/iconRowLabel`, which is a `'use client'` component. Exporting the path constant from the root barrel is fine, but the README says the `Icon` server component is on a separate subpath to avoid dragging server-only imports into client bundles. The inverse risk exists here: the root barrel exports `formatSVGHook` / `formatSvg` (which have `'use server'` directive), meaning any client-bundle import of `IconLabelPath` or `AtomicIcon` from the root will drag the `'use server'`-tagged hook module along. **Fix:** Remove `formatSVGHook` and `formatSvg` from the root barrel export (`src/index.ts:6`) or split them into a separate subpath (e.g. `./hooks/formatSVG`).

- [ ] _Reviewed ‚Äî not an XSS vector: `pt.c.da` is typed `Record<string, string>` and is built by atomic's `SSRProps` purely from `data-*` keys (`data-${key}`), which are inert presentation attributes ‚Äî spreading them can't introduce `onClick`/`dangerouslySetInnerHTML`. No directive is needed (it inherits its parent's server boundary and the JSX runtime supplies React). Left as-is._ **`svgChild/component.tsx` missing `'use server'` / `'use client'` directive and React import** ‚Äî `src/blocks/svgChild/component.tsx:1-4` ‚Äî The file has no directive and no `import React from 'react'` (relies on the JSX runtime transform). The component uses `{...pt?.c?.da}` spread which passes `data-*` attributes down but there is no type safety on what `da` contains ‚Äî a malicious or incorrect `da` value could pass `onClick` or `dangerouslySetInnerHTML` via the spread. **Fix:** Validate or type-narrow `pt?.c?.da` to only allow safe SVG presentation attributes before spreading.

- [x] **`iconSet` versions config has `maxPerDoc: 50` but no `purgeOnChange` or explicit retention strategy** ‚Äî _Documented: added a comment on the `versions` config noting that with `schedulePublish` + `validate` every draft save makes a version, so a busy set silently drops history past 50, and that the number trades audit depth against storage._ ‚Äî `src/collections/iconSet.ts:249` ‚Äî With `schedulePublish: true` and `validate: true`, each draft save creates a version. With `maxPerDoc: 50`, busy icon sets will silently drop old versions once the limit is hit. This may surprise users auditing change history. **Fix:** Document the retention limit in the JSDoc or README, or lower it to a more conservative default with a comment.

- [ ] _Reviewed ‚Äî same conclusion as the `da` spread above: `contents` is rendered inside the SVGChild via atomic's `SSRProps`, which puts it on `c.p.dangerouslySetInnerHTML` for the `<svg>` body and only ever emits `data-*` strings into `c.da` ‚Äî it does not flow user content into event handlers. The `svgChild`/`SVGChild` path is admin-authored content (authenticated), consistent with the `icon` collection's posture. Left as-is._ **`svgChild` block: `contents` field is `textarea` with no sanitization** ‚Äî `src/blocks/svgChild/block.ts:38` ‚Äî Raw SVG path data/content is entered by a CMS user and stored verbatim. `svgChild/component.tsx` spreads it via `pt?.c?.da`, which bypasses the `dangerouslySetInnerHTML` surface but may still expose XSS if `da` contains event handlers. ## Notes

- **Decision (read access left auth-gated):** The frontend renders icons via the cached `svgString` inlined server-side (`Icon.tsx`, `iconChild/component.tsx`), never via the public `/api/icon/file` URL. Guests therefore never need file read access, and keeping `read: authd` matches the project's "assets are not public" posture (same as `@pro-laico/fonts`). The High finding is intentionally not changed.
- **Decision (exports/blocks ‚Üí `src`):** `.`, `./Icon`, and the block subpaths pointing at `src/` is intentional for monorepo/`transpilePackages` consumption, not a bug. The committed `dist/` (admin subpaths) was rebuilt to match the edited source.
- The `svgString` code field (`src/collections/icon.ts:103-106`) has no `access` restriction, so any authenticated user who can read the `icon` collection can retrieve the raw (optimized) SVG markup ‚Äî this is likely intentional but worth confirming.
- The `Icon` collection has no `timestamps: false` override, so Payload will add `createdAt`/`updatedAt` (correct default behavior for an upload collection).
- The `iconsPlugin` correctly handles the `onInit` composition gap (it does not call `onInit` at all, which is fine because neither collection registers one), but if a future version adds `onInit`, the pattern of `config.onInit ? [config.onInit, newInit] : [newInit]` should be used.
- `createIconSelect` is not exported from the root barrel (correctly), but the `./admin/iconSelect` export subpath in `package.json` points to `dist/` while `./admin/iconRowLabel` also points to `dist/` ‚Äî these are consistent and correct, unlike the main entry points.

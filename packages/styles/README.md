# @pro-laico/styles

> The single home for all CSS handling in Atomic Payload: the `designSet` + `shortcutSet` collections (the Tailwind-config equivalent authored inside Payload), the token fields and admin row-labels, and the runtime UnoCSS processor (`createCssProcessor` + `processDesignSet`) that turns the active design set into CSS for both the frontend and Payload's admin.

## What this package is

A Payload plugin + runtime that owns:

- **`designSet`** — the full theme document. Tabs for colors, sizing, screens, animations, and arbitrary design tokens. The active design set drives every class generated by the runtime CSS processor.
- **`shortcutSet`** — reusable UnoCSS shortcuts. Equivalent to writing a Tailwind component class once and applying it everywhere. Optional but almost always wanted.
- **`createCssProcessor`** — the UnoCSS pipeline that reads the active design set + shortcut set + stored atomic classes and writes generated CSS to the storage globals.
- **`processDesignSet`** — the `beforeChange` pass that compiles a `designSet` document's tokens, colors, prose styles, and preflights into the storage fields the CSS processor consumes.

Plus the helpers and admin UI that make those collections feel native: value/token fields, row-label components, and a small set of path constants for Payload's import map.

## Why it exists

Atomic Payload's whole pitch is "build websites in the dashboard, no code." That only works if the admin user can edit the equivalent of a Tailwind config — design tokens, color scales, breakpoints — without touching a file. `designSet` is that file as a Payload document. Because only one is active at a time, you can build alternative designs as draft documents and swap which one renders by changing the active flag.

The split between `designSet` and `shortcutSet` mirrors UnoCSS's split between theme and shortcuts: tokens vs. reusable class compositions.

Centralizing the CSS processor + `processDesignSet` here means every piece that has to know about CSS — token shape, preflight generation, UnoCSS presets — lives in one package.

## Quick start

```ts
import { buildConfig } from 'payload'
import { stylesPlugin } from '@pro-laico/styles'

export default buildConfig({
  plugins: [
    stylesPlugin({
      atomicHook: yourBeforeChangeHook,         // from @pro-laico/atomic/hook
      generateLivePreviewPath: yourPreviewUrl,  // optional
      // getCached: yourGetCached,              // attaches the standalone cssHook (CSS without @pro-laico/atomic)
      designSet: {
        // false (or { enabled: false }) to omit the designSet collection entirely
        // access, collection — shallow-merged per-collection settings
      },
      shortcutSet: {
        // false (or { enabled: false }) to omit shortcutSet entirely
        // defaultShortcuts: [...]   read-only rows shown in admin
        // access, collection — same pattern as designSet
      },
    }),
  ],
})
```

Row-label and admin component paths default to template-provided `@/ui/...` strings resolved via your app's `importMap`.

## Concepts to know

- **Active design set.** The CSS processor reads the currently active design set on every change and re-generates CSS from it. Switching active sets is how you swap an entire design.
- **Tokens vs shortcuts.** Tokens are values (colors, sizes); shortcuts are class compositions. Both feed into the same UnoCSS pipeline downstream.
- **Protected names.** `shortcutProtectedNames` is the set of identifiers the runtime reserves. Show this list in admin UX when accepting user-defined shortcut names.

## Where it sits in the monorepo

Depends on `core`, `site`, `zap`. Consumed by `atomic/hook` (whose `beforeChange` wraps `createCssProcessor` + `processDesignSet`) and `seed` (which writes default sets).

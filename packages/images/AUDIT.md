# @pro-laico/images — Payload Plugin Audit

## Summary

The published `dist/` tree was compiled from a materially different version of the source than what lives in `src/` — it includes a `blurDataUrls` option, a `require()` call, and imports from packages (`@pro-laico/ap-utils`, `@pro-laico/ap-apf`) that no longer exist in this monorepo. The `exports` map and `main`/`types` fields all point at raw TypeScript source files rather than compiled output, so consumers either get broken imports or rely on path-aliasing that may not generalise. Several correctness issues exist in the plugin's override logic, the Favicons upload config, cache-hook placement, and the component's alt-text handling.

## Findings

> **Status:** Stale-`dist`, override deep-merge, `beforeChange`→`afterChange` revalidation (+ new `afterDelete`), required `alt`, dangling favicon `adminThumbnail`, `image/jpg` dupe, and admin `useAsTitle`/`defaultColumns`/favicon label all fixed (dist rebuilt). Source-only `exports` and `blocks/**` build-exclusion are intentional for workspace consumption. README blur-data-urls wording, `FaviconField` deepMerge arg-order, imageChild `alt` UX, the redundant `./src` alias, and the `getCached`/`@payload-config` note are now all fixed; dist re-built clean. Nothing outstanding beyond the intentional source-only packaging.

### 🔴 Critical

- [x] **Dist built from a different source revision — stale package references** — _Fixed: clean-rebuilt; dist now imports only `@pro-laico/core` + `payload` (verified no `ap-utils`/`ap-apf` refs)._ — `dist/collections/images.js:2` imports `revalidateCacheCollection` from `'@pro-laico/ap-utils'`; `dist/fields/favicon.js:1` imports `onUploadSetAPF` from `'@pro-laico/ap-apf'`. Neither package exists in the monorepo. The published dist will fail at runtime the moment any consumer uses the `images` or `FaviconField` exports from the compiled tree. **Fix:** Run `pnpm build` after all source is reconciled; verify `dist/` imports only `@pro-laico/core` and `payload`.

- [ ] _Intentional: source-only workspace consumption — every export resolves to `src`, the same decision applied across `@pro-laico/*`. The committed `dist` exists only for a hypothetical npm publish and is rebuilt clean._ **`exports` map and `main`/`types` point to raw TypeScript source** — `package.json:10-32`. Every entry (`"."`, `"./schema"`, `"./src"`, `"./blocks/imageChild"`, `"./blocks/imageChild/component"`) resolves to `.ts`/`.tsx` files. Bundlers that do not transpile `node_modules` (e.g. a standard Node `require`) will break. **Fix:** Add `import`/`require`/`types` conditions pointing to `./dist/...`; keep the `default` condition as a source-path fallback only if this is an intentional source-only monorepo pattern, and document it explicitly.

- [x] **`src/plugin.ts` and `dist/plugin.js` are out of sync** — _Fixed by the rebuild: `dist/plugin.{js,d.ts}` is now generated from `src/plugin.ts`, so the phantom `blurDataUrls`/`blurOptions` no longer appear in the compiled output or its types._ — `src/plugin.ts:35-45` has no `blurDataUrls`/`blurOptions` options and no `require('@oversightstudio/blur-data-urls')` call; `dist/plugin.js:4-28` does. `dist/plugin.d.ts:16-22` exposes `blurDataUrls` and `blurOptions` in the public interface. Any consumer reading the type declarations will configure options that the source-built version ignores entirely. **Fix:** Decide the canonical implementation (README favours the external-plugin pattern in `src/`), delete the divergent path, rebuild.

### 🟠 High

- [ ] **`blocks/**` excluded from the build** — _Intentional: the block subpaths resolve to `src` (workspace consumption), same as the rest of the exports map._ — `tsconfig.build.json:11` has `"exclude": ["dist", "node_modules", "src/blocks/**"]`. The `imageChild` block and its component are never compiled to `dist/`, yet `package.json:25-32` maps `./blocks/imageChild` and `./blocks/imageChild/component` through the exports map. Those subpaths resolve to raw `.ts`/`.tsx` source files. **Fix:** Remove `src/blocks/**` from the `exclude` list in `tsconfig.build.json`, or add a dedicated build step that compiles the blocks, and update the exports map to point to the compiled output.

- [x] **`imagesOverride`/`faviconsOverride` shallow-merges destructively** — _Fixed: a shared `mergeCollection` deep-merges `upload`/`access`/`admin`, appends `fields`, and merges `hooks`._ — `src/plugin.ts:41-42`. A spread `{ ...Images, ...imagesOverride }` replaces entire top-level keys, so passing `imagesOverride: { fields: [...] }` silently discards the `alt` field, and `imagesOverride: { upload: { focalPoint: false } }` discards all `imageSizes`, `mimeTypes`, `formatOptions`, and `adminThumbnail`. The comment says "Shallow override" but no JSDoc warns of the array-replacement hazard. **Fix:** Either deep-merge `upload` and `fields` sub-keys, or document the destructive semantics clearly and rename the options to `imagesReplace`/`faviconsReplace`.

- [x] **`revalidateCacheCollection` placed on `beforeChange`, revalidates stale data** — _Fixed: new `revalidateCacheCollectionAfterChange` (core) runs post-commit; Images now uses it._ — `src/collections/images.ts:13`. The hook fires before Payload writes the new document. `DEFAULT_REVALIDATION_HANDLERS.images` (`core/src/hooks/collection/revalidate.ts:62-64`) calls `revalidateTag('image', originalDoc?.id)`, which invalidates the cache while the new URL hasn't been persisted yet. A simultaneous request re-populates the cache with the old document. **Fix:** Move the hook to `afterChange` (use `CollectionAfterChangeHook`), where `doc` carries the committed state.

- [x] **Favicons `adminThumbnail: 'thumbnail'` references a size that does not exist** — _Fixed: removed `adminThumbnail` (Payload falls back to the original)._ — `src/collections/favicons.ts:11`. The `favicons` upload config declares no `imageSizes`, so Payload never generates a `thumbnail` variant. The admin list view will show broken image placeholders. **Fix:** Either remove `adminThumbnail` (Payload will fall back to the original) or add a `imageSizes` entry named `thumbnail` with appropriate dimensions for `.ico` files.

- [x] **README claims blur-data-URLs is wired automatically by the plugin** — _Fixed: README now states the plugin does **not** auto-wire it and points to the `imagesPlugin` JSDoc wire-it-yourself snippet (intro line, `src/` table row, and peer-deps note all updated)._ — `README.md:14`: "If `@oversightstudio/blur-data-urls` is installed, the plugin opts the Images collection into automatic blur placeholder generation." The current `src/plugin.ts` does not do this — it explicitly documents that the consumer must wire the plugin themselves. **Fix:** Align README with source: remove or rewrite that sentence to match the current "wire it yourself" instruction in the JSDoc.

### 🟡 Medium

- [x] **`alt` field on `Images` collection is not required** — _Fixed: `required: true` (the seed doesn't populate this collection, so no data break)._ — `src/collections/images.ts:12`. Alt text is the primary accessibility attribute for images. An unrequired field means editors can (and will) upload images without alt text, causing downstream a11y failures when `imageChild` falls back to the collection's `alt` value. **Fix:** Add `required: true` to the `alt` field, or if intentionally optional, add `admin: { description: 'Required for accessibility.' }` and enforce it in the `imageChild` block instead.

- [x] **`imageChild` block `alt` field is not required and falls back silently to `''`** — _Fixed: the block `alt` field description now explicitly tells editors to leave it blank only for decorative images (non-decorative images need alt text for accessibility)._ — `src/blocks/imageChild/component.tsx:8`. The component does `const { alt = '', ... } = pt?.c?.p || {}` — an empty string is a valid `alt` (marks image as decorative) but it is applied unconditionally even for non-decorative images. The block field at `src/blocks/imageChild/block.ts:61` is also not required. Editors get no signal that omitting alt text makes the image inaccessible. **Fix:** Mark the block `alt` field with `admin: { description: 'Required for non-decorative images.' }` at minimum; consider a conditional `required` when the `fill` or `priority` checkboxes are checked.

- [x] **No `afterDelete` hook on the `Images` collection** — _Fixed: added `afterDelete: [revalidateCacheOnDelete]` + an `images` entry in `DEFAULT_DELETE_REVALIDATION_HANDLERS`._ — `src/collections/images.ts:13`. `revalidateCacheCollection` only revalidates on change. Deleting an image leaves stale cache entries that will render broken `<Image>` tags until natural expiry. `DEFAULT_DELETE_REVALIDATION_HANDLERS` in core (`core/src/hooks/collection/revalidate.ts:71`) has no `images` entry either. **Fix:** Add `afterDelete: [revalidateCacheOnDelete]` to `Images.hooks` and add an `images` handler to `DEFAULT_DELETE_REVALIDATION_HANDLERS` in core.

- [x] **`deepMerge` argument order in `FaviconField` allows callers to stomp base config** — _Fixed: `type: 'upload'` and `relationTo: 'favicons'` are re-asserted after the merge so a caller can't silently re-target the picker; `name`/`admin`/`hooks` remain overridable (real usage in `site` overrides `name`+`admin`)._ — `src/fields/favicon.ts:34`. `deepMerge(faviconField, rest)` merges `rest` (caller args) into `faviconField` (base), which means caller values win over base values — including `name`, `type`, and `relationTo`. A caller passing `{ relationTo: 'images' }` silently re-targets the field to the wrong collection. **Fix:** If caller overrides should be limited to `admin`/`hooks` sub-keys, validate and reject `name`/`type`/`relationTo` overrides. If full override is intentional, document it.

- [x] **`dist/index.js` does not re-export `ImagesPluginOptions`** — _Fixed by the clean rebuild from reconciled source._ — `dist/index.js:1-5` has no `export type { ImagesPluginOptions }`, but `src/index.ts:4` does. The type is exposed in `dist/index.d.ts:2` via the declarations, but it comes from `./plugin.d.ts` which carries the divergent interface (includes `blurDataUrls`). **Fix:** Rebuild from the reconciled source; verify the exported type matches `src/plugin.ts`.

- [x] **`image/jpeg` and `image/jpg` both listed in `mimeTypes`** — _Fixed: removed the non-standard `image/jpg`._ — `src/collections/images.ts:18`. `image/jpg` is not an official MIME type; the IANA-registered type is `image/jpeg`. Including both is harmless but creates confusion — `image/jpg` will never match a real browser-reported MIME type. **Fix:** Remove `'image/jpg'` from the array.

### 🟢 Low / nice-to-have

- [x] **`Images` collection missing `admin.useAsTitle`** — _Fixed: `useAsTitle: 'alt'`._ — `src/collections/images.ts:11`. Without it, Payload falls back to `id` in the relationship UI. Setting `useAsTitle: 'alt'` would make relationship pickers and list views show meaningful labels. **Fix:** Add `useAsTitle: 'alt'` to `admin`.

- [x] **`Images` collection missing `admin.defaultColumns`** — _Fixed: `defaultColumns: ['alt', 'updatedAt']`._ — `src/collections/images.ts:11`. Default list columns are inferred by Payload. Specifying `defaultColumns: ['alt', 'updatedAt']` (or similar) gives editors a more useful table view immediately. **Fix:** Add `defaultColumns` to `admin`.

- [x] **`Favicons` collection has `fields: []`** — _Fixed: added a required `label` text field + `useAsTitle: 'label'`._ — `src/collections/favicons.ts:9`. An empty fields array is valid, but an `alt` or `label` text field would help editors identify which favicon variant a document represents (e.g. light/dark mode). **Fix:** Add at minimum a `{ name: 'label', type: 'text', required: true }` field and set `admin.useAsTitle: 'label'`.

- [x] **`getCached` in component called with positional args that depend on host config** — _Fixed: added an inline note documenting that `@pro-laico/core/cache/auto` binds to the host app's `@payload-config`, so the component only works inside a Next.js app that resolves that alias._ — `src/blocks/imageChild/component.tsx:16`. `getCached('image', block?.image?.id, block?.version)` uses the `auto` variant from `@pro-laico/core/cache/auto`, which is bound to the host project's `@payload-config` via a module-level `cache()` call. This works only if the consuming Next.js app resolves the same `@payload-config` alias. The dependency is invisible — no import of `payload.config` is visible here. This is an architectural note rather than a bug, but it means the component cannot be used outside the standard template. **Fix:** Document this assumption in the component or README.

- [x] **`src/` subpath in `exports` map is redundant** — _Fixed: removed the `"./src"` alias (pure duplicate of `.`); no consumer referenced it._ — `package.json:20-23`. `"./src"` maps to the same file as `"."`. If the intent is to give consumers direct access to all source files, this doesn't achieve it (it's a single-file alias). If it's a mistake, remove it.

- [x] **`loading: 'lazy'` has a `defaultValue` in block config but `decoding` does not** — _Fixed: `decoding` now `defaultValue: 'async'`._ — `src/blocks/imageChild/block.ts:73-76`. Both fields default to `lazy`/`async` per the descriptions, but `decoding` has no `defaultValue: 'async'`. Editors who save without touching `decoding` get `null` in the document, and the component receives `undefined`. **Fix:** Add `defaultValue: 'async'` to the `decoding` select field.

## Notes

- The dist tree should be treated as a build artifact and should not be manually edited. Re-building from reconciled source will resolve all dist-specific findings.
- The `@pro-laico/ap-utils` and `@pro-laico/ap-apf` package references in dist indicate an older monorepo layout; search other packages' dist trees for the same stale references before the next publish.
- The `getCachedImage` function in core does a `findByID` without `depth: 0` or a `select` clause — it fetches the full document to extract only a URL. This is a core-package concern but worth noting here since `images` consumes it.

title: "@pro-laico/seed"
description: "Fill a fresh project with sample content in one click: a SEED DATABASE button in the admin and a guarded POST /api/seed endpoint behind it."
---

`@pro-laico/seed` gives you a one-click way to populate a brand-new project with the content that makes Atomic Payload feel alive: a starter design set, a default icon set, header and footer, sample pages, and site metadata. It adds a **SEED DATABASE** button to the admin dashboard and the guarded `POST /api/seed` endpoint behind it, so you can go from an empty database to a working site without writing any setup data by hand.

<Callout type="info">
  `@pro-laico/seed` is a Tool package, a building block of the Atomic Payload stack, meant to be used alongside `@pro-laico/core` and the other `@pro-laico/*` packages (most easily via the [atomic-payload template](/docs/templates/atomic-payload)), not on its own.
</Callout>

<Callout type="warn">
  Seeding is **destructive**: it clears the collections it manages and recreates them from scratch. Run it on a fresh project, not over content you want to keep. By default the endpoint allows **any authenticated user** to run it, so lock it down with the [`authorize`](#options) option in multi-user or production apps.
</Callout>

## Setup

<Tabs items={['Standalone', 'With the atomic-payload template']}>

<Tab value="Standalone">

Adding seeding to your own Payload + Next.js project.

<Steps>

<Step>
### Add the plugin to your Payload config

With nothing passed, the plugin uses the bundled Atomic Payload seed. It's common to gate it behind an environment variable so it only registers where you want it:

```ts
import { buildConfig } from 'payload'
import { seedPlugin } from '@pro-laico/seed'

export default buildConfig({
  plugins: [
    seedPlugin({
      enabled: process.env.INCLUDE_SEED === 'true',
    }),
  ],
})
```

This does two things:

- Mounts a `POST /api/seed` endpoint that runs the seed function. It's guarded: it returns `403` unless the request comes from an authenticated user (and, when you pass an [`authorize`](#options) predicate, unless that predicate also approves them).
- Adds a **`BeforeDashboard`** banner to the admin dashboard with a **SEED DATABASE** button that POSTs to that endpoint and shows a toast as it runs.
</Step>

<Step>
### Register the admin button in the import map

The dashboard banner is an admin React component that Payload loads through its import map. After enabling the plugin, regenerate the map so Payload can find it:

```bash
pnpm payload generate:importmap
```

Without this step the endpoint still works, but the **SEED DATABASE** button won't appear on the dashboard. (If you'd rather not show the button at all, set `includeBeforeDashboard: false` and call the endpoint yourself.)
</Step>

<Step>
### Lock it down for production

Out of the box the endpoint allows any authenticated user to seed. That's fine for a solo project, but risky once other people have logins. Pass an `authorize` predicate to restrict who may run the destructive seed:

```ts
seedPlugin({
  enabled: process.env.INCLUDE_SEED === 'true',
  authorize: (user) => user.roles?.includes('admin') ?? false,
})
```

The predicate runs after Payload's auth check and receives the authenticated user; return `true` to allow the seed.
</Step>

<Step>
### Run the seed

Open the admin dashboard and click **SEED DATABASE** in the welcome banner. The button POSTs to `/api/seed`, the bundled seed clears and recreates its collections, and a toast confirms when it's done. Your site then has content to render.
</Step>

</Steps>

</Tab>

<Tab value="With the atomic-payload template">

The [atomic-payload template](/docs/templates/atomic-payload) already wires everything up: the plugin is in the config and the admin button is in the import map, so the **SEED DATABASE** button is ready on the dashboard.

<Steps>

<Step>
### Click SEED DATABASE

Open the admin dashboard. The welcome banner has a **SEED DATABASE** button; click it to populate a fresh project with the starter design set, icon set, header/footer, sample pages, and site metadata. A toast confirms when it finishes, and you can then visit the site to see the content.
</Step>

<Step>
### Lock it down for production

The template registers the plugin as a plain `seedPlugin()`, so seeding is always available. Because the seed is destructive, pass `enabled: false` (or an `authorize` predicate) in `src/plugins/index.ts` once your project has real content you don't want overwritten.

```ts
seedPlugin({ enabled: false })
```
</Step>

</Steps>

</Tab>

## Options

`seedPlugin(options?)` accepts:

<TypeTable
  type={{
    enabled: { type: 'boolean', default: 'true', description: 'Set false to turn the plugin off, so it adds neither the endpoint nor the admin button.' },
    seed: { type: 'SeedFn', default: 'bundled seed', description: 'The function the endpoint runs. Defaults to the bundled Atomic Payload seed; pass your own to seed different content. To reuse the bundled seed with different slugs, wrap it: `(args) => seed(args, { pages: "my-pages" })`. See the slug keys below.' },
    endpointPath: { type: 'string', default: "'/seed'", description: 'Path mounted under /api, so the default resolves to /api/seed.' },
    authorize: { type: 'SeedAuthorize', default: 'any authenticated user', description: 'Predicate run after the auth check to further restrict who may seed. Receives the authenticated user and returns `boolean | Promise<boolean>`. Because the seed is destructive, pass a role test in production, e.g. `(user) => user.roles?.includes("admin")`.' },
    includeBeforeDashboard: { type: 'boolean', default: 'true', description: 'Whether to add the SEED DATABASE banner to the admin dashboard. Set false to keep the endpoint but hide the button.' },
  }}
/>

The `seed` function accepts an optional second argument, `slugConfig` (a `SeedSlugConfig`), for remapping which collection and global slugs the bundled seed writes to. It isn't a plugin option of its own; you reach it by wrapping the `seed` option (above). Every key is optional and defaults to the slug the Atomic Payload collections use:

<Accordions>

<Accordion title="SeedSlugConfig keys">

<TypeTable
  type={{
    pages: { type: 'string', default: "'pages'", description: 'The pages collection the sample pages (home, prose showcase, testing, not-found) are created in.' },
    header: { type: 'string', default: "'header'", description: 'The header collection the seeded site header is written to.' },
    footer: { type: 'string', default: "'footer'", description: 'The footer collection the seeded site footer is written to.' },
    forms: { type: 'string', default: "'forms'", description: 'The forms collection the bundled backend form is created in.' },
    icon: { type: 'string', default: "'icon'", description: 'The icon collection the seeded SVG icons (logo, menu, check, close, theme, cookie, github) are uploaded to.' },
    iconSet: { type: 'string', default: "'iconSet'", description: 'The iconSet collection the starter icon set is written to.' },
    designSet: { type: 'string', default: "'designSet'", description: 'The designSet collection the starter design set (theme tokens) is written to.' },
    shortcutSet: { type: 'string', default: "'shortcutSet'", description: 'The shortcutSet collection the starter shortcuts are written to.' },
    siteMetaDataGlobal: { type: 'string', default: "'siteMetaData'", description: 'The global the seeded site metadata is written to.' },
  }}
/>

</Accordion>

</Accordions>

<Callout type="info">
  See [Slugs](/docs/concepts/slugs) for how those collections and globals are addressed across the stack.
</Callout>

All options at their defaults, as a working starting point:

```ts
import { seedPlugin } from '@pro-laico/seed'

seedPlugin({
  enabled: true,
  // seed defaults to the bundled Atomic Payload seed; pass your own SeedFn to change it
  endpointPath: '/seed',
  // authorize defaults to "any authenticated user"; pass a predicate to restrict further
  includeBeforeDashboard: true,
})
```

## Environment variables

The plugin reads no environment variables. Control registration with the `enabled` option instead: pass `false` (or gate it behind your own env var) to skip the endpoint and the admin button.
